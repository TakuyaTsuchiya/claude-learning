## S4-V1 eラーニング（構成案ドラフト）

### 提案タイトル
現状維持「【人事向け】社内eラーニングシステムを自作しよう」

### ゴール
社内研修動画を管理・配信するeラーニングシステムをHTML単品で自作し、月額LMS代を削減しながら**自社に最適な進捗管理・動画再生画面**を手に入れるようになる。架空企業「ビジネスカレッジ」の新人研修4本（導入・営業基礎・商品知識・データ活用）を例に、インタラクティブなHTMLシステムを10分で構築・運用までを一気通貫で実演する。

### 重要トピック
- **LMS月額費用の現実**（日常の「サブスク疲れ」と同じ：100名の組織なら年300万円超。カスタマイズできず、要件に合わない機能が満載）
- **HTMLなら1ファイル、コストはゼロ、自社要件に最適な研修プラットフォームが手に入る**（既存Excelスプレッドシートを学習進捗DBに転用、YouTube埋め込みで配信、LocalStorageで個人の再生位置を保存）
- **「難しいシステム」から「シンプルなHTMLのテンプレート」へ**（日常の「テンプレート」「ひな形」の発想：一度作れば何度も再利用、カスタマイズも簡単）
- **S1の「ドメイン知識のプロダクト化」の実証例**（人事のドメイン知識＝「研修ローテーション」「習熟度判定」「受講完了の証明」を仕様書なしにHTMLで形にする）
- **中立的な比較**：LMS有償版は「規模が大きい組織向け」、自作は「カスタマイズ重視の組織向け」。どちらが優れているではなく、組織規模と要件次第
- **属人化の防止**（人事担当者が研修システムの構築・運用スキルを得ると、外部ツール依存から脱却）
- **次セクション（S5）への布石**（HTMLのみで完結するS4 vs DBやバックエンドを伴うS5の違いを暗示）

### 受講者のつまずき
1. 「eラーニングシステムは高額な専門ツールが必要」という思い込み
2. 「HTMLと動的機能を両立させるには複雑なバックエンドが必要」だと誤解（実際はJavaScript＋LocalStorageで十分）
3. 「架空企業での例では自社運用が想像つかない」（デモの最後に「あなたの研修4本をこのHTMLに入れ替えるだけ」の置き換えやすさを強調）

### 画面操作方針
スライドは表紙のみ。画面収録で実演。
- 人事部門の研修管理課題を背景に、従来LMS一般論を説明（個人情報配慮で実商品名は避ける）
- `demos/elearning-html/` を開く → コース一覧（4本のカード配置）→ 各コースをクリック → 動画再生画面（YouTube埋め込み）＋進捗バー（JavaScript自動更新）
- 「全視聴完了」ボタン → LocalStorage自動保存、DevToolsで`localStorage`内容を一瞥
- 別ウィンドウで別アカウントをシミュレート（進捗データの独立性を確認）
- HTMLソースコードの主要部分を簡潔に眺め（10行程度）「JavaScriptはたった数十行で、DBもサーバも不要」
- **個人情報配慮**：実企業の研修題目・受講者名は使わず、架空企業「ビジネスカレッジ」「営業基礎」「データ活用」など汎用名で統一

### 位置付け方針
S4「業務効率化の実践」の「HTMLレポート単品」の延長。S1-V2で学んだ「ドメイン知識をプロダクト化する」思想の実証例として、人事のドメイン知識（研修ローテーション・進捗管理・完了証明）をコード不要なHTMLで実装。S3（Skills・MCP）のような自動化・連携とは異なり「テンプレート化されたツール」という位置付け。次セクションS5では「DB・バックエンド・デプロイ」が出てくるため、S4-V1は「ローカルファイル・ブラウザストレージで完結」の限界を体験させ、「次は何が必要か」への自然な流れを作る。

### 参照一覧
- `demos/elearning-html/index.html`（架空ビジネスカレッジの新人研修4本）
- YouTube埋め込みコード（公開動画なら外部リンク可）
- JavaScript localStorage API：`https://developer.mozilla.org/ja/docs/Web/API/Window/localStorage`
- S1-V2「HTMLレポート」との接続確認
- S5との対比用メモ（「HTMLのみ vs DB＋バックエンド」）

### シーンの流れ（9シーン）
まず**人事部門が抱える研修管理の苦労を宣言**する。新人研修4本を全新入社員に受けさせ、完了状況を把握し、証明書を発行する。従来はLMS月額を払い、毎月「誰が何を終わったか」をLMS管理画面で確認している。ところが機能が多すぎて使いこなせず、カスタマイズもできず、月額が無駄になっていないか。50〜60秒で「なぜ重要か」を腰を据えて語る → 次にS1-V2「HTMLレポートは自作できる」との接続を一言入れる。同じ発想で「eラーニングシステムもHTML一枚で自作できる」ことを示す → 題材として架空「ビジネスカレッジ」の新人研修4本を紹介 → `demos/elearning-html/` を開く、コース一覧（4カード）が表示 → 各コースをクリックし動画再生画面（YouTube埋め込み＋進捗バー＋最終視聴日時）に遷移 → 「全視聴完了」ボタンでLocalStorage自動保存、DevToolsで内容確認 → 別ウィンドウで別アカウントをシミュレート、進捗データが独立していることを確認 → HTMLソースコードの主要部分を眺め「JavaScriptはたった数十行で、DBもサーバも不要」という驚きを共有、同時に「ブラウザを閉じたら進捗は消える」という限界も正直に述べ、「規模が大きくなったらバックエンドが必要」とS5への布石 → クロージングで「あなたの研修4本をこのHTMLに入れ替えるだけで、月額コストゼロの研修プラットフォームが完成」と実行可能性を強調

### 設計ノート
- **既知題材の再利用**：S1-V2「HTMLレポート単品」思想を研修システムという別ドメインに適用
- **対比構造**：月額LMS vs 自作HTML を「組織規模と要件で使い分け」と中立化
- **S5への布石**：「HTMLのみで完結」の限界体験→「複数ユーザデータ永続化にはDB・バックエンドが必要」
- **LocalStorage実演**でブラウザだけで動く驚き

## S4-V2 請求書・Excel（構成案ドラフト）

### 提案タイトル
現状維持「【経理向け】毎月の請求書・Excel処理を自動化しよう」

### ゴール
顧客データCSVから請求書PDFを自動生成し、さらにExcelシートの請求処理・入金管理を一括自動化するツール群を作れるようになる。バックオフィス特有の「データ形式の統一・手入力ミス・手作業の繰り返し」という3大課題をClaude Codeで解決する実感を得る。アウトプットは実運用レベルの請求書テンプレート（PDF化対応）と、月次請求一覧・入金実績・未払い抽出を自動更新するExcelワークシート。

### 重要トピック
- **バックオフィス業務の「毎月の定型作業」が最高の自動化対象**（営業や企画と違い、経理は「同じフォーマット・同じチェック項目」の繰り返し）
- **「請求書は紙でなくデータ」という発想の転換**（日本の商慣習では紙という認識が強いが、デジタルファースト企業はデータベース→PDF→デジタル保存で効率化）
- **PDF生成は標準ライブラリで一発**（jsPDF・pptxgenjsなどJavaScript、またはPython openpyxl・reportlabで実装可能。「複雑な印刷ツールが必要」の思い込みを払拭）
- **Excelとの連携が経理実務の実際**（請求書生成だけでなく、売上台帳・入金管理・消費税集計までワンツール化）
- **入力ミス・二重請求・未入金フォローの自動化が実務価値**（日常の「エクセル関数」「ピボットテーブル」の延長線）
- **小規模企業（年商数億以下）こそ自動化でキャッシュフロー改善に直結**（大企業のERPに比べ低コスト・すぐ運用開始）
- **架空データで実感・実装・試験まで完結**（受講者が自社データに置き換える想定）

### 受講者のつまずき
1. 「PDF生成は難しそう」という先入観（jsPDF／openpyxlで数十行のコードで完成）
2. 「Excelは手作業が正義」というマインドセット（自動化で抜け漏れが増える不安、踏み出せない）
3. 「複数ファイル・複数形式の連携が複雑では？」（CSV→JSON→PDF→Excelの一連の流れをClaude Codeのマルチステップで簡潔に完結できる）

### 画面操作方針
スライドは表紙のみ。画面収録で実演。
- 題材データ提示（顧客マスタCSV、請求内容JSON）
- Claude Codeで「請求書PDFテンプレート作成」プロンプト（jsPDF選択、reportlabは口頭補足）
- PDF生成・ブラウザ表示（顧客名・請求金額・支払期限の動的挿入確認）
- Excel処理実演（openpyxl＋pandasで請求一覧・入金台帳を自動更新）
- 請求書PDF→Excel台帳→未払い抽出レポート、一気通貫
- 「毎月、新しい顧客データをCSVで放り込むだけで請求書10件分が自動生成」という運用イメージ
- **個人情報配慮**：実在企業・実在顧客データは使わず、架空建築会社（仮名：○△建設）の仮想取引先。金額・氏名・住所すべて架空

### 位置付け方針
S4「職種別・業務効率化」の経理実務編。S3-V2（Skills Creatorで月次レポートSkill化）の応用として、「経理業務の中でレポートと請求書という2大エッセンシャル業務の両方を自動化」する戦略を示す。S4-V1（eラーニング）は学習LMS（HTMLレポート単品）だったのに対し、V2は「バックオフィス組織全体に波及する基礎ツール」でスコープを拡張。V3（オンボーディングCSV）への伏線として「CSV入力→複数形式出力」の設計パターンを共通化。

### 参照一覧
- 架空企業データセット：`demos/kensetsu-invoice/`
  - `customers.csv`（顧客マスタ）
  - `invoices.json`（請求内容）
  - `payments.xlsx`（入金実績）
- PDF生成ライブラリ：jsPDF（ブラウザベース）／reportlab（Python低レベル）／openpyxl（Excel生成兼）
- Excel処理：pandas（CSV→DataFrame→Excel）／openpyxl（セル単位制御）
- S3-V2「Skills Creator」との連携言及

### シーンの流れ（11シーン）
まず**バックオフィス経理の「毎月やってくる地獄」を宣言**。経理担当者は月初に50社分の顧客データを見直し、1件ずつ請求書を手作業で作成。「100万円のはずが101万円」という手入力ミスが月1〜2件発生、営業から「この請求書、金額合わんで」と返される。一方Excelには売上台帳があるはずなのに、請求書とデータが分散し「あの顧客、まだ入金されてない?」の確認に30分。50〜60秒で腰を据えて語り、「顧客マスタに新規顧客を追加するだけで請求書が勝手に生成される」と価値提示 → 次にS3-V2（Skills Creator）月次レポート自動化と並行する「経理の2大柱」の位置付けを明示（V2はレポート化、S4-V2は請求書＋収金管理、両者で経理業務の属人化を解消） → 題材「○△建設（仮名）」の月間請求業務を紹介（顧客マスタCSV100社、請求金額JSON、請求書PDF50件一括生成＋Excel入金台帳更新） → Claude Codeで請求書PDF自動生成プロンプト、jsPDFかreportlabかの選択肢を口頭補足、今回はjsPDFで進める → コード生成の流れ（CSV読み込み→JSONパース→テンプレート定義→ループで生成）、実際に顧客5社分のPDFをブラウザ確認 → Excel処理に移り、openpyxl＋pandasで売上台帳に請求内容追記、「入金実績」マスタと照合して「未払い一覧」を自動抽出 → 未払い先に「送付後30日超」「金額100万超」条件フィルタ、営業への督促リストCSVも自動生成 → 「毎月、新規顧客と新規請求をファイルに放り込むだけで、請求書＋入金台帳＋未払いリスト全部更新」の運用イメージ → 「属人化の解消＝ベテラン経理のコツが組織ツール化」の意味を明示 → クロージング：「バックオフィスは定型作業の最大密度エリア、月50時間のコストが5時間に圧縮」とまとめ、S4-V3へのつながりを示唆

### 設計ノート
- **S3-V2との関係性**：V2は「経営判断材料を高速化」、S4-V2は「業務プロセス自体を削減」
- **スコープ**：jsPDF／openpyxl等の標準ライブラリ＋Claude Code実装、DB・サーバー・デプロイなし
- **ライブラリ選択**：「組織の技術スタック・共有手段による」と中立化
- **つまずき対策**：「PDF複雑では」→最初は3項目のシンプルテンプレートで達成感、「Excel二重入力」→CSV→Excel自動書き込みで手作業0、「毎月の微調整」→ヘッダ1行編集だけ

## S4-V3 オンボーディングCSV（構成案ドラフト）

### 提案タイトル
新提案「新入社員CSVで部署別チェックリストを自動化 — 人事の『毎回手作業』をHTML1枚に」

### ゴール
新入社員の入社前〜初日〜初月の受け入れ業務を、**部署別に異なる研修・システムアクセス・備品配置のチェックリストをCSVから自動生成**し、人事の手作業タスク管理から解放される。アウトプットはHTML形式で、営業・エンジニア・バックオフィスといった部署ごとに「やることリスト」を見た瞬間に理解できる形で完成させる。架空の中小企業の新入社員データ（営業2名・エンジニア2名・経理1名）を使い、一気通貫でチェックリストが出来上がる様子を実演する。

### 重要トピック
- **「どの会社にもある業務」だからこそ自動化の価値がある**（人事業務は業種・規模を選ばず、新人受け入れは必ず発生）／日常の「やることリスト」「チェックシート」との接続

## S4-V4 週次サマリー／S3 MCP応用編（構成案ドラフト）

### 重要トピック

## S4-V5 LINE Harness（構成案ドラフト）

## S4-V7 スクレイピング・アタックリスト（構成案ドラフト）

### 提案タイトル
新提案「【営業向け】Webスクレイピングでアタックリスト自動生成 — 手作業8時間が数分になる理由」

### ゴール
企業事例ページのURLから営業アタックリスト（対象企業・部署・役職・氏名）を自動生成でき、スクレイピングという技法の「2段階実行」（一覧ページ→個別ページ）と「LLMが自律的にURLを抽出する仕組み」を実務に活かせるようになる。アウトプットはCSV形式のアタックリスト。

### 重要トピック
- **アタックリストとは＝営業が狙う相手先企業の構図図面**（営業現場の共通言語：新規営業・インサイドセールス・BtoB事業開発が毎日参照、誰が・どの企業の・どの部署まで見つけるかで受注確度が変わる）
- **手作業100社8時間 vs Claude Code数分＝時間差の正体は「人間が読む作業を自動化」**（手作業が悪ではなく、その時間を営業戦略・顧客関係構築に使える）
- **2段階スクレイピングの構図**：Webは階層化（一覧ページ→個別ページ）、一覧クロール→個別URL抽出→個別ページから情報採掘、URLパターン理解で精度が劇的に上がる
- **LLMが自律的にURLを抽出する理由**：Claude Codeは文脈理解が可能（「このページにケーススタディリンクあり、パターンは `/case/`」）、人間のドメイン知識（「URLにcase, customers等が含まれやすい」）をプロンプトで渡すと判定精度が上がる
- **robots.txt・利用規約の確認**：スクレイピング前に対象サイトの規約を読み、許可の有無を確認する責任
- **Sansan事例ページを題材にする理由**：公開情報・スクレイピング許可・営業向け実践的データ構造の3点で最適
- **CSV出力→ローカル便利ツール化→S5でSaaS化への予告**：今回はCSV出力のローカルツール、次セクションでSaaS化（Webインターフェース＋チーム共有）への質的転換

### 受講者のつまずき
1. **「アタックリストの業務価値がピンとこない」**→冒頭で手作業100社のタイムラプスを見せ「8時間が数分になったら営業はどう変わるか」の人間ドリブン価値提示
2. **「スクレイピングって難しそう、URLを個別に渡さないと」**→実演で「URLひとつ（一覧）を渡すだけでLLMが個別URLを抽出して掘り下げる」を見せる
3. **「CSVの中身が合っているか、確認方法がわからない」**→出力CSVをExcelで開き、Sansan事例ページを手作業で確認して「一致している」ことを視覚検証

### 画面操作方針
スライドなし。ターミナル画面のみ（画面収録）。
- robots.txt確認（`curl https://jp.sansan.com/robots.txt | grep -i case`）
- Claude Codeプロンプト：Sansan事例ページ一覧URL入力→一覧クロール→企業ロゴURL抽出（URLパターン`case`, `cases`, `customers` 条件）→個別ページスクレイピング→会社名・部署・役職・氏名をJSON→CSV出力
- 出力ファイル（`attack_list.csv`）をExcelで開いて内容確認
- Sansan公式ページをブラウザで複数ケース確認、値が合致することを検証

### 位置付け方針
S4（業務効率化の実践）の締めくくり動画。S4-V1〜V6は「単機能の効率化」だったが、V7は「複雑なデータ取得→加工→出力」という**多段階ワークフロー**を一気通貫で実装する難度。S4セクション全体の「小さな局所最適化」から「事業全体を視野に入れた戦略的自動化」への卒業地点。S5「アプリのフロントエンドを開発しよう」への橋渡し：「今はローカルCSV出力」だが「次はWebアプリ化（HTML入力フォーム→データベース→チーム共有）」という質的転換を予告。

### 設計ノート
- **冒頭50〜60秒の価値宣言**：手作業100社8時間→数分の時間差をリアル描写
