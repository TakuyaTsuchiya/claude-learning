# Claude Code実践講座 プロジェクトルール

## WBS・ガントチャート管理
- WBS・ガントチャートの唯一の情報源は `docs/curriculum-review.html`（HTMLレビューサイト）
- タスクの追加・削除・変更はすべてこのファイルに対して行う
- `docs/gantt-chart.md` は使わない

## 制作パイプライン全体像

### フロー
レビューサイト（構成＋台本）→ スプレッドシートに転記 → カリキュラム進捗管理シート → 収録・編集 → LMS公開

### 1. レビューサイト（`docs/curriculum-review.html`）— Single Source of Truth
- 役割：各レクチャーの**構成と台本**を一元管理する唯一の情報源
- 構成に必要な7項目（これが揃えば台本が書ける）：
  1. **ゴール** — 見終わった受講者が何ができるようになるか
  2. **重要トピック** — カバーする内容
  3. **受講者のつまずき** — 予想される詰まりポイント
  4. **画面操作方針** — デモあり/なし、何を見せるか
  5. **位置付け方針** — 導入/実践/まとめ等
  6. **参照一覧** — 使う資料・CSV・URL・デモファイル等
  7. **シーンの流れ** — 話題の順序（「まず〜／次に〜／最後に〜」の話し言葉で繋ぐ。Step番号は使わない）
- 台本は `docs/scripts-data.js` に記述し、`curriculum-review.html` から動的レンダリング（関心の分離）
  - 構成7項目はHTML側（`curriculum-review.html`）に直書き
  - 台本本体は `scripts-data.js` の `SCRIPTS` オブジェクトに追加
  - 1エントリの構造：
    - `meta`：duration / mode / goal
    - `materials[]`：type / name / purpose / timing
    - `script[]`：start / end / topic / direction / content / reference

### 台本作成ワークフロー（実演デモ講座向け）
- 土屋が Screen Studio で画面収録＋ラフ解説
- Screen Studio のトランスクリプト機能で文字起こし
- Claude Code（/daihon Skill）で川端さん用台本に整形
- 川端さんがナレーション再収録
- 編集チームが映像＋ナレーションを合成

### 台本執筆スタイル（川端さんナレーション用）

#### トーン：中級者向けの落ち着いた口語体
- カジュアル過ぎず、硬すぎない。中級者が一度で腹落ちするペース
- 川端さん口調「〜ですね」「〜しましょう」「やってみます」を基調
- 押し付け表現（「〜ですよね」）は避け、「〜と思います」など柔らかく

#### 推奨語と避ける語
- 「やります」→「扱います／進めていきます」
- 「投げる」→「依頼する」
- 「なるんです」→「になります／ようになります」
- 「でも」→「一方で」
- 「そんなの無理」→「現実には不可能」
- 「突っ込む」→「着手する」
- 「丈夫さ」→「堅牢さ」

#### 見出しの作法
- 「Step 1：」「Step 2：」の**見出し番号は使わない**
- 「まず〜」「次に〜」「最後に〜」で話し言葉として繋ぐ
- 構成7項目の「シーンの流れ」も同じ作法で書く

#### 中立化（断罪表現を避ける）
- 手法比較（例：Vibe Coding vs Spec-Driving）は**優劣ではなく「コンテキスト量の差」「用途の使い分け」**で整理
- 「落とし穴」→「限界」
- 「破綻します」→「狙い通りが出にくくなります」
- 「ダメ」「悪い」は避け、特性で説明

#### 専門用語と日常語の橋渡し
- カタカナ専門用語（例：「スペック」）は**日常での使われ方（例：「パソコンのスペック」）との接続**を1文で入れる
- 英語略称は**英語フルスペル＋日本語訳**を添える（例：spec = specification = 仕様書）

#### 論理構造
- 冒頭シーンで**「なぜ扱うのか／メリットは何か」**を腰を据えて語る（目安50〜60秒）
- 同じメッセージを複数シーンで繰り返さない（冒頭で宣言→後続シーンは定義・実演に絞る）

#### 題材スコープの階層
- **S1・S2（入門）**：HTMLレポート単品レベル。レビュー・テスト・動作検証は扱わない
- **S5・S6（アプリ開発）**：DB・CI/CD・デプロイ・テスト戦略まで扱う

### 2. 台本スプレッドシート（Googleスプレッドシート）
- 役割：レビューサイトからの**転記先**。編集チーム・音声担当への配布用
- レビューサイトの台本と内容は同一。スプレッドシート側で独自に書き換えない
- 実例：M5セクション1動画1

### 3. カリキュラム進捗管理シート（Googleスプレッドシート）
- 役割：動画編集チームとの**制作進捗管理**
- カリキュラム進捗管理シートの列構造（カテゴリ別）：
  - **動画メタ情報**: コース名 / 時間（分） / 画面収録（分） / 動画タイトル / 動画の説明文 / 需要度 / 重要度 / 難易度 / セキュリティ / 職種メインカテゴリ / クイズ内容 / 講師担当
  - **担当者**: ヒルウラ担当 / 外部編集担当 / 音声担当 / FB優先順位 / プロマネ
  - **ステータス管理**: ステータス / FB / ステータス更新日時 / Youtubeリンク
  - **素材データ**: 修正用データ / スライド / アバター映像 / 画面収録 / ヒルウラ編集後データ（mp3）
  - **編集パイプライン**: 編集着手 → 初稿完了予定 → 黒河FB → スライドデータ準備 → 音声+アバター準備 → 画面収録データ準備 → BSデータ準備 → 編集着手 → 納品 → 黒河FB → 川端FB → 最終版提出 → 川端最終チェック → LMS公開予定日
  - **台本・資料**: 担当 / A-E列完了 / 台本完了 / スライド数
  - **撮影**: 担当 / ステータス / 成果物
  - **最終チェック**: 担当 / 動画時間 / 動画本数 / 成果物
  - **進捗フラグ**: 目次 / 資料作成 / 台本 / 撮影 / 編集 / チェック / 納品
  - **備考**: 音声データ不備 / 川端FB反映有無
