## S6-V1 名刺管理アプリにDB接続して永続化（構成案・確定版）

### 提案タイトル
「名刺管理アプリにDB接続して永続化しよう」（20字）

### ゴール
前のセクションで作った名刺管理アプリのローカルストレージ保存をSupabase接続に置き換え、「ブラウザを閉じても別端末からも同じデータが見える」永続化を実装できるようになる（APIキーと`.env`の扱いもここで初めて実体験する）。

### 重要トピック
- **ローカルストレージとクラウドDBの対比**（ローカル＝自分のブラウザだけに残るメモ、クラウドDB＝自宅のファイルをクラウドに上げる感覚。どちらが優劣ではなく、用途の違いで選び分ける）
- **テーブル設計＝Excelの列を決める感覚**（contacts テーブル1枚、列は id / name / company / email / phone / created_at の6列。Excelの1行目に列名を並べる作業と同じ）
- **CRUD4つの操作**（Create／Read／Update／Delete、Excelで「行を足す／表を見る／セルを直す／行を消す」の4操作に対応）
- **APIキーと`.env`の実際の扱い**（前のセクションで布石を置いた「詳しい扱いはこの先で」の回収。`.env`に `SUPABASE_URL` と `SUPABASE_ANON_KEY` を記述、`.gitignore`で除外、Vercelの環境変数画面にも同じ値を登録。家の鍵を写真に撮ってSNSに載せない感覚と同じ）
- **Supabaseクライアントの呼び出し**（`createClient(url, key)`、実演時はコマンド連鎖で一気通貫）
- **「ブラウザ閉じても消えない」体験**（画面を閉じる→別タブで開き直す→スマホでも同じデータが見える、感覚として「自分のブラウザを超えた」ことを体感）

### 受講者のつまずき
1. **`.env`と環境変数という言葉で止まる**（中身は「鍵を1箇所にまとめたテキストファイル」だけ、実演でファイルを開いて中身を見せれば腑に落ちる）
2. **テーブル設計を何から決めればいいか迷う**（アプリで入力している項目をそのまま列にするだけ、名刺なら「名前・会社・メール・電話」がそのまま列になる）
3. **ローカル保存の処理をどこまで消せばいいか分からない**（読み書きの関数を1〜2箇所置き換えるだけで、画面のコードはほぼ触らない）

### 画面操作方針
スライド比率は全体の2割（表紙＋「ローカル保存 vs クラウドDB」対比図＋contactsテーブル設計の1枚のみ）、画面収録比率は8割。APIキーの扱い：実演時のみANON KEYを表示、撮影後に必ずローテーション、動画内で「撮影後に無効化します」と一言添える。
- 実演順：Supabase管理画面でcontactsテーブル作成（Table Editorから6列定義）→ Settings→APIからURLとANON KEYをコピー → ローカルで`.env`を新規作成 → `.gitignore`に`.env`が入っていることを確認 → `npm install @supabase/supabase-js` → Claude Codeに「ローカルストレージの読み書きをSupabaseに置き換えて」と依頼 → ローカル動作確認（名刺を1件登録）→ Supabase管理画面のTable Editorで行が増えたことを確認 → GitHub push → Vercel環境変数画面にURLとキーを登録 → 自動デプロイ → 本番URLをスマホで開き、同じデータが見えることを確認
- 架空の名刺データ（例：架空商事の山田太郎）を使い、実在企業・個人名は避ける

### 位置付け方針
S6の起点であり、前のセクションで作った3つのアプリを「自分のブラウザだけの試作」から「複数端末で共有できる本物のアプリ」に昇格させる最初の1本。ここで覚える流れ（テーブル作成→`.env`設定→Claude Codeに置き換え依頼→環境変数の本番登録→デプロイ）は、続く経費アプリ・CRMでも同じ順序で反復するため、3本シリーズの**手本回**として丁寧に見せる。APIキーと`.env`は前のセクションの安全回とSupabaseセットアップ回の両方で布石を置いていた話題で、ここで初めて実際のファイル操作として回収する。認証・権限管理・本番運用の論点はこの3本では扱わず、後のBACKSTAGE社の回で「プロダクトとして公開するときの別の層」として改めて整理する。

### 参照一覧
- 前のセクションで作った名刺管理アプリのリポジトリ（GitHub上、ローカルストレージ保存版）
- Supabase公式 `https://supabase.com/`
- Supabase JS SDK公式 `https://supabase.com/docs/reference/javascript/introduction`
- `.env`サンプル（`SUPABASE_URL=...` / `SUPABASE_ANON_KEY=...` の2行）
- Vercel 環境変数設定画面 `https://vercel.com/docs/projects/environment-variables`
- 前のセクションの安全回（APIキーの扱いはこの先で実体験すると布石を置いた回）
- 前のセクションのSupabaseセットアップ回（管理画面ツアーで「APIキーの場所」を確認した回）

### シーンの流れ（10シーン）
まず冒頭で「前のセクションで3つのアプリを作りました。見た目は本物のSaaSでも、データは自分のブラウザの中だけ。ブラウザを閉じるとスマホでは何も見えないし、同僚とも共有できません。今日はその最後の壁を取り払います。名刺管理アプリにクラウドのデータベースを繋いで、『ブラウザを閉じても、別の端末でも、同じデータが見える』状態にします」と宣言（50〜60秒）→ 次にローカル保存とクラウドDBの対比を1枚のスライドで整理（ローカル＝冷蔵庫の付箋、クラウドDB＝自宅のファイルをクラウドに上げる感覚、優劣ではなく用途の使い分け）→ テーブル設計はExcelの列を決める感覚、と日常語で橋渡ししてcontactsテーブル1枚の設計を提示（id／name／company／email／phone／created_atの6列）→ Supabase管理画面のTable Editorでcontactsテーブルを作成（実演）→ 続いてプロジェクトのSettings→APIからURLとANON KEYをコピーし、ローカルに`.env`を新規作成して2行書く（「この1ファイルがすべての鍵、だから`.gitignore`に入っていることを必ず確認」と強調）→ `npm install @supabase/supabase-js`でSDKを追加、Claude Codeに「ローカルストレージの読み書きをSupabaseに置き換えて」と依頼（CRUDの4操作が一度に置き換わる様子を見せる）→ ローカルで名刺を1件登録し、Supabase管理画面のTable Editorで同じ行が増えたことを確認（「ブラウザの外側にデータが出た」瞬間）→ GitHubにpushし、Vercelの環境変数画面にURLとキーを同じように登録して本番デプロイ（`.env`はGitに上がらない、環境変数は本番環境にも別経路で渡す、という構造を口頭で整理）→ 本番URLをスマホで開き、さっきPCで登録した名刺がそのまま見えることを確認（「ブラウザを閉じても消えない」体験の達成点）→ 最後に「これで名刺アプリはローカル試作からクラウド本物に昇格しました。同じ流れを経費アプリ・CRMでも反復します。認証や権限管理はこの3本では扱わず、BACKSTAGE社の回でまとめて整理します」と次回予告で閉じる

### 設計ノート
- **対比構造**：「ローカルストレージ vs クラウドDB」を冒頭で明示し、同じ対比をシーン末尾（スマホで同じデータが見える）で回収
- **伏線回収**：前のセクションで置いた「APIキー・`.env`の扱いはこの先で」を、`.env`作成／`.gitignore`確認／Vercel環境変数登録の3点セットで実演レベルまで落とす
- **リズム配置**：5分／10シーン、冒頭宣言に50〜60秒、残り9シーンは平均25〜30秒
- **APIキーのガードレール**：実演時のみキー表示、撮影後にローテーション、動画内で「撮影後に無効化します」と一言
- **3本シリーズの手本回**：V2・V3は同じ順序（テーブル作成→`.env`→置き換え依頼→環境変数登録→デプロイ）を踏襲するため、V1では各ステップの「なぜこの順序か」も軽く触れて型を共有
- **スコープ厳守**：認証／RLS／複数テーブル／JOIN／集計クエリには触れない、「永続化だけ」に絞る

## S6-V2 経費アプリにDB接続して永続化（構成案・確定版）

### 提案タイトル
「経費アプリにDB接続して永続化しよう」（18字）

### ゴール
フロント完結の経費アプリにSupabaseを接続し、経費データを端末を超えて残せるようにする。2回目の反復で「APIキー取得 → テーブル作成 → CRUD実装」という永続化の型を体得し、次に別ドメインのアプリでも同じ手順で接続できる自信を持ち帰る。

### 重要トピック
- **2回目だからリズムを上げる**（前の動画で名刺管理アプリを永続化したときに踏んだ3ステップ（APIキー・テーブル・CRUD）を、経費アプリで同じ順に辿り直す。同じ手順が別ドメインで通用することが「再現性」の体感になる）
- **別ドメインで同じ型が通用する実感**（扱うデータは日付・金額・カテゴリ・メモと名刺とは別物だが、永続化の型は共通。家計簿アプリでも住所録でも、同じExcelに書式だけ変えて記録するのと同じ感覚）
- **テーブル設計の小さな拡張**（`expenses`（id / date / amount / category / memo / created_at）の単一テーブル。金額は数値型、日付は日付型、カテゴリは文字列型と、列ごとに型を選ぶ発想を軽く押さえる）
- **ローカルストレージ → Supabase の置き換え**（既存のローカル保存ロジックを差し替えるだけで、コードの骨格は変わらない。自分の引き出しのメモ帳を、チーム共有のクラウドに移すイメージ）
- **本講座では永続化の型に集中する判断**（集計クエリ（合計・カテゴリ別の平均など）も組み合わせれば実装できるが、本講座ではシンプルな永続化の型を揺るぎないものにする方を優先。集計は受講者が独り立ちしてから自分の題材で試してもらう領域として整理）
- **レシート画像はブラウザ表示のみに留める**（クラウドストレージに預ける選択肢もあるが、今回は永続化の骨格を通す回として表示のみに絞る）
- **認証・権限管理は後半に集約**（誰のデータかを区別する仕組みはBACKSTAGE社の回で扱う、このV2では「データが端末を超えて残る」までをゴール）

### 受講者のつまずき
1. **「同じ手順でもう一度やる意味が見えない」**（単なる繰り返しに見えると飽きる、冒頭で「2回目は手が覚えに行くフェーズ」と宣言し、速度を上げることで学習の進捗感を担保）
2. **「金額や日付の列の型って何を選ぶの？」**（テキスト型しか触ったことがない受講者向けに、数値型・日付型を選ぶ基準を1行で示す。金額は計算したくなるから数値型、日付は並び替えたくなるから日付型）
3. **「集計グラフを外して見栄えが寂しくなるのでは」**（フロント側のグラフ描画はそのまま残し、データ取得先だけDBに差し替える構成、永続化だけを差し替えたことが明確に伝わるよう完成画面で残存グラフを示す）

### 画面操作方針
スライド比率は低め（表紙＋完成画面キャプチャのみ、尺の1割以下）、残り9割は画面収録の一気通貫。実演順：Supabase管理画面を開く → `expenses` テーブルを作成（列：id / date / amount / category / memo / created_at、型はそれぞれ選ぶ）→ APIキーを `.env` に貼る（前の動画と同じ流儀で手早く）→ Claude Code に「ローカルストレージ保存を Supabase の `expenses` テーブルに置き換えて」と依頼 → 生成差分を確認 → ブラウザで経費を登録 → Supabase管理画面で行が増えたことを確認 → ブラウザを閉じて開き直しても経費が残っていることを確認 → 別タブで同じURLを開いても同じデータが見えることで永続化を体感。個人情報配慮として、経費データは架空の内容（会議費／交通費／消耗品費など）に統一し、実在の店舗名・取引先名は使わない。

### 位置付け方針
S6の反復定着回。前の動画で永続化の型を初めて通した受講者に、同じ型が別ドメインでも通用することを体感してもらう位置。ここで型が定着すれば、次の動画でCRMを永続化するときには手が先に動く状態に近づく。S6全体の学習曲線としては「1回目で型を学ぶ → 2回目で速度を上げて定着 → 3回目で手が覚える」の中央に置かれる回であり、受講者の自信がもっとも伸びる瞬間をここに設計する。BACKSTAGE社の回で認証・権限管理に踏み込む前に、シンプルな永続化の骨格を3つのアプリで揃えきる流れの中継点。

### 参照一覧
- 前のセクションで作ったfreee風経費アプリ（ローカルストレージ版、置き換え元のコード）
- Supabase公式ドキュメント：`https://supabase.com/docs`
- Supabase JavaScript クライアント：`https://supabase.com/docs/reference/javascript/introduction`
- 前のセクションのSupabaseセットアップ回（APIキーの場所と管理画面の位置が既出）
- 前の動画（V1）で整備した `.env` とSupabaseクライアント初期化パターン
- 本プロジェクトのCLAUDE.md（台本スタイル・中立化ルール）

### シーンの流れ（9シーン）
まず冒頭で「前の動画で名刺管理アプリを永続化しました。今日は同じ手順を経費アプリで辿り直します。2回目なので手が覚えにいくフェーズです。完成すると、経費データが端末を超えて残るようになります」と宣言 → 次に今日の完成形を30秒プレビュー（経費を登録 → ブラウザを閉じる → 開き直しても残っている）→ Supabase管理画面で `expenses` テーブルを作成（列と型を1列ずつ選ぶ、金額は数値型・日付は日付型という選び方を短く添える）→ APIキーを `.env` に貼る（ここは前の動画と同じ流儀で手早く通す）→ Claude Code に「ローカルストレージ保存を Supabase の `expenses` テーブルに置き換えて」と依頼し、生成された差分を確認 → ブラウザで架空の経費を登録し、Supabase管理画面で行が増えたことを確認 → ブラウザを閉じて開き直し、経費が残っていることを体感（ここで「型が定着した」ことを言語化）→ 集計クエリにも踏み込める選択肢があるが本講座では永続化の骨格に集中すると整理し、レシート画像のクラウド保存も同じ理由で次の機会に回すと伝える → 最後に「次の動画では同じ型をCRMで3回目に辿ります。そこまで来れば永続化の型は自分の型になっています」と予告してクロージング

### 設計ノート
- **速度の上げ方**：APIキー取得とテーブル作成は「前回と同じ手順なので短く」と言い切り、尺を削る。代わりに「ブラウザを閉じて開き直しても残っている」の体感シーンに時間を残す
- **リズム配置**：9シーンで4分、冒頭宣言とプレビューに約50〜60秒、テーブル作成とAPIキーに約60秒、CRUD実装と動作確認に約90秒、クロージングに約30秒
- **シンプル化の判断の中立化**：集計クエリ／レシート画像のクラウド保存は「できないから外す」ではなく「永続化の型を揺るぎなくするためにここでは扱わない」と位置付ける
- **2回目の体感設計**：前の動画と同じ言葉（「APIキー」「テーブル」「CRUD」）を意図的に同じ順で繰り返し、聞き覚えの効果で手順の定着を促す
- **グラフは残す**：フロント側のグラフ描画はそのままにし、データ取得元だけをローカルストレージから Supabase に差し替える
- **個人情報の扱い**：経費データはすべて架空（会議費／交通費／消耗品費などの一般名）
- **動画間の接続言及**：「前の動画で名刺管理を永続化した」「次の動画ではCRMで同じことを反復する」の大枠言及に留め、具体的なV番号は本文に書かない

## S6-V3 CRMアプリにDB接続して永続化（構成案・確定版）

### 提案タイトル
「CRMアプリにDB接続して永続化しよう」（18字）

### ゴール
前のセクションで作ったSalesforce風CRMにSupabaseを接続し、顧客データと商談ステータスをクラウドで永続化できるようになる。DB接続3回目の反復を通じて「もう自分で好きなSaaSのDB接続ができる」という独り立ち感覚を確立する。

### 重要トピック

### シーンの流れ（10〜11シーン、枠のみ）
まず冒頭50〜60秒で「前の動画でBACKSTAGE社の日常を見ました、今回は本番運用編です。あなたはここまででDB接続アプリを作れるようになった、プロはここから先をこう運用しています」と講座最終動画としての宣言と高揚感の土台を置く → 次にBACKSTAGE社の本番運用事例の全体像を俯瞰（[要先方確認]）→ 「あなたのプロトタイプ vs プロの本番」境界線図解で受講者の現在地を可視化 → 認証・権限管理の実装例を中立的に紹介（[要先方確認]）→ 脆弱性対策と定期スキャンの日常運用（[要先方確認]）→ 監視・ログ・インシデント対応の文化（[要先方確認]）→ CI/CD・テスト戦略の自動検品ライン（[要先方確認]）→ スケーラビリティと運用体制（[要先方確認]）→ 受講後の次のステップ案内（コミュニティ・専門書・エンジニアとの協業・継続学習）→ 講座全体（S1〜S6）の振り返りと達成感の確認 → 最後に「プロもAIで武装しています、あなたもその入口に立ちました」と高揚感あるクロージング

**注記**：上記シーンの具体内容は先方確認後に詳細化。最終シーンのクロージングは講座全体の締めとして最重要、受講者が前向きに講座を閉じられる語り口を台本段階で磨き込む。

### 確定待ち事項（先方確認が必要）
- 撮影方法（インタビュー／密着／オフィス画面収録／ナレーション差し込み）
- 出演者（及川さん中心か、他のエンジニアも登場するか）

### 設計ノート
- BACKSTAGE社の協力範囲確認後に詳細化する枠ドラフト
- 旧V4「セキュリティ注記」5項目（認証・権限／脆弱性対策／監視・ログ／スケール／運用体制）を吸収し、注意喚起ではなく「プロの実例」として前向きに展開
- 講座最終動画として、達成感・独り立ち・次のステップを全部入れる、受講者を萎縮させない設計を最優先
- 「あなたのプロトタイプ vs プロの本番」境界線図解は必ず入れる、受講者の現在地を肯定しつつ、本格運用との差を中立的に可視化
- 中立化（断罪回避）の徹底：「落とし穴」→「境界」、「破綻します」→「狙い通りが出にくくなります」、「ダメ」→「用途が異なる」で統一
- 動画間の接続は番号指定を避け、「前の動画で日常編を見た、今回は本番運用編」「ここまでで作ったプロトタイプ」と役割と現在地で言及
- 先方確認が進み次第、シーンごとの具体内容・素材パス・台本本体（`scripts-data.js`）へ展開
