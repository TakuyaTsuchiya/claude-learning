# S5/S6 ゼロベース再構成 — 設計書

## 背景と動機

Goodpatch社のClaude Code全社導入事例（185人中86%がデプロイ到達）を受け、カリキュラムにSupabaseを追加。S5/S6をゼロベースで再構成し、GitHub/Vercel/Supabaseを使ったアプリ開発トラックとする。

SalesNowのCLAUDE.md記事を参考に、アプリ開発向けCLAUDE.mdの書き方も講座内で扱う。

## 設計方針

### 確定事項

| 項目 | 決定 |
|---|---|
| 構成 | S5: 6本、S6: 6本（現行7本→6本に変更） |
| パターン | 1動画1アプリ（S3/S4と同じ） |
| 導入動画 | なし。V1からアプリを作りながら学ぶ（「概念より先に動きを見せる」ポリシー準拠） |
| S5の位置付け | GitHub + Vercel + スペック駆動。フロントのみ。自分用業務ツール |
| S6の位置付け | Supabase + プロの開発フロー。DB付き。チーム/顧客向けアプリ |
| 職種別 | S3/S4と同じく、動画ごとに異なる職種に刺さるユースケース |
| 進化ギミック | S5-V1→S6-V2（商談メモ帳→CRM）、S5-V4→S6-V3（経費チェッカー→チーム経費精算）の2本 |
| プロ要素 | 独立した概念説明動画は作らず、アプリを作る過程に自然に織り込む |
| BACKSTAGE社事例 | S6-V1に独立1本。内容は先方に任せる |

### S5とS6の切り分け

- **S5 = 「自分ひとりで使う業務ツール」** — フロントのみ。ローカルストレージ/静的サイト。GitHub + Vercelで管理・公開
- **S6 = 「チームや顧客と共有するアプリ」** — Supabase（DB + 認証）付き。データ永続化・複数ユーザー対応

S5-V1の「ローカルストレージだとPC変えたら消える」→ S6-V2で「Supabaseで解決」という伏線回収により、Supabase導入の動機が自然に生まれる。
