## S3-V2 Skills Creator（構成案ドラフト）※経理マン月次レポート題材

### 提案タイトル
「毎月30時間の月次レポート作成をSkill化する — Skills Creatorで自分の業務を手放す」

### ゴール
経理担当者が毎月繰り返している**月次経営報告レポート作成**を題材に、Skills Creatorで自分の業務をSkill化し、**同じ品質のレポートをチーム誰もが生成できる状態**にできるようになる。アウトプットは経営者向けHTMLインフォグラフィックレポート。ドレミファエア（架空の航空会社）の3月度月次レポート作成までを一気通貫で実演する。

### 重要トピック
- **Skill化の対象＝「毎月・毎週・毎日、同じ手順で繰り返す業務」**（日常の「マニュアル」「レシピ」と同じ発想で、手順を書き出せば誰でも同じ品質で再現できる）
- **Skills Creatorの正体＝「自分の作業手順をClaudeに説明するだけでSkill仕様に変換される対話型ツール」**（プロンプトエンジニアリングの知識は不要、口頭で「何をやっているか」を話せばいい）
- **S3-V1との対比**：V1は「税理士が作ったshinkoku（他者のSkill）を使う」体験、V2は「自分が作り手になる」体験。受動→能動への転換
- **非エンジニア業務こそSkill化の主戦場**：経理・人事・営業・マーケの「毎月やってくる定型業務」が最も効く。コードが書けなくてもSkillは作れる
- **「数字だけのレポート」から「読まれるレポート」へ**：月次業績CSV＋議事録＋事業メモをインプットに、背景・ストーリー・示唆を含むHTMLレポートを自動生成（参考：経理AI実践ガイドなど書籍の指針）
- **属人化の解消**：ベテラン経理のノウハウをSkill化＝組織全員が同じ品質で月次レポートを出せる資産に
- **小さく始める戦術**：完璧なSkillを一発で作らず、まず1ヶ月分を回してテスト→微調整→本番運用の流れ

### 受講者のつまずき
1. 「何をSkill化すればいいかわからない」：業務全体を丸ごとSkill化しようとして取っ掛かりを失う（実は「毎月必ずやる定型業務」が最適な候補）
2. 「自分の作業を言葉で説明できる自信がない」：手順が頭の中で暗黙知になっていて、書き出せない。Skills Creatorの対話形式がこのハードルを解決する
3. 「Skillの品質は1回で完成するの?」：テスト実行と微調整を繰り返す前提を知らず、完璧を求めて踏み出せない

### 画面操作方針
スライドは表紙のみ。画面収録で実演。一気通貫の流れ：
- 業務の洗い出し（経理マンの1ヶ月業務リストを提示→「月次経営報告」を選ぶ判断軸）
- Skills Creator起動（`/skill-creator`）→ 対話形式で手順を口頭説明 → `SKILL.md` が自動生成される過程
- 生成されたSkillの中身（ロール・インプット・アウトプット条件）を確認
- テスト実行：`demos/doremifa-air/` の3ファイル（financials.csv・board-notes.md・business-notes.md）をSkillに渡し、HTMLレポートを生成
- ブラウザでレポート確認（業績サマリー・背景・ストーリー・経営への示唆）
- 微調整（「グラフが足りない」「示唆が浅い」といった気付きを対話で反映）
- チーム共有への言及（口頭で「このSkillを共有すればチームで同じ品質のレポートが出せる」とひと言触れるのみ、別人登場なし）
- **個人情報配慮**：実在企業のIRデータは使わず、架空のドレミファエア社データで統一。受講者が自社データに置き換える想定で説明

### 位置付け方針
S3の核となる「作り手体験」。V1（他者作のshinkokuを使う＝受動）の直後に置き、「使って仕組みが分かった」受講者を「自分が設計する側」へ反転させる。経理という非エンジニア業務を題材にすることで、**コードが書けない受講者でもSkillの作り手になれる**ことを実証する。同時にV3（MCP）への伏線として「Skillだけでは外部ツール（Slack・Notion等）には触れない」→「次はそれを繋ぐMCP」へ繋ぐ。

- S3-V1 shinkoku実演動画（対比用）

- 参考書籍（引用なし・発想の参照のみ）：経理AI実践ガイド ケース08「読まれるレポート作成」
- 出力例：ドレミファエア 2026年3月度 月次経営レポート（HTMLインフォグラフィック）

### シーンの流れ（10シーン）
まず**毎月やってくる月次レポート作成の憂鬱を宣言**する。経理担当者は月末に業績データをかき集め、議事録を読み直し、パワポにグラフを貼り、増減理由を書き込む。その結果出てくるのは「数字の羅列」で、経営会議で読まれないまま終わる。毎月10〜30時間がこの作業に消える。ここを50〜60秒で腰を据えて語り、「もしこの業務がSkillになっていたら、毎月の負担はゼロに近づく」と価値提示する。次に、S3-V1で体験したshinkokuとの対比を置く。V1は「税理士が作ってくれたSkillを使う」体験だったが、V2は「自分が作り手になる」体験。受動から能動への転換を明示する。題材として「ドレミファエアの3月度月次経営レポート」を紹介。架空の航空会社で、ドレミファソラシドの7路線を持つ中堅キャリア。業績CSV・経営会議メモ・事業背景メモの3ファイルをインプットに、経営者向けHTMLレポートを出力する、というゴールを設定する。Skills Creatorを起動し（`/skill-creator`）、「あなたの業務手順を教えてください」という対話に答える形で、経理担当者としての手順を口頭で話していく。業績数値を読み込んで前年比・前月比を計算する、議事録から経営陣の関心事を抽出する、事業メモから新路線や競合動向といった定性情報を拾う、これらを合わせて数字の背景にあるストーリーを組み立てる、最後にHTMLインフォグラフィックとして出力する、という流れを口語で説明する。この対話の応答が `SKILL.md` に自動変換されていく過程を見せる。生成されたSkillの中身を確認し、ロール・インプット・アウトプット条件の構造を視覚化する。テスト実行に移り、`demos/doremifa-air/` の3ファイルを渡して実際にレポートを生成。ブラウザで開いて内容を確認すると、数字の羅列ではなく「LCC参入で価格競争懸念」「新路線シ就航で四国需要の取り込み」といった背景付きの読まれるレポートが出てくる。ここで微調整が必要な箇所（例：「グラフが弱い」「示唆をもう1段踏み込みたい」）を対話で反映し、Skillを育てていく様子を見せる。完成したSkillについて「このまま共有すれば他の経理メンバーも同じ品質で月次レポートを出せる」とひと言触れる。クロージングでは「Skillは手順の外部化＝組織の資産」と位置付けてまとめる。

### 設計ノート
- **対比構造**：V1（他者作を使う・受動）→ V2（自分が作る・能動）
- **非エンジニア題材**：経理マンの月次レポートで「コード書けなくてもSkillは作れる」を実証
- **経理系の連続性**：V1 shinkoku（確定申告＝個人経理）→ V2 月次レポート（法人経理）でS3の世界観にまとまりを作る
- **書籍引用の扱い**：経理AI実践ガイド ケース08の思想（「読まれるレポート」）は発想の参照として1文触れる程度。プロンプトの直接引用はしない（著作権配慮）
- **次セクション布石**：「Skillは単体ツール内で完結」→「外部連携はMCP」でV3へ接続

## S3-V3 Slack MCP（構成案ドラフト）

### ゴール
Claude CodeでSlack MCPを接続し、全チャンネルを横断して日報・週報を自動生成できるようになり、S3で初めて学ぶ「MCPの定義」と「外部ツール連携の本質」を理解する。

### 重要トピック
- **MCPとは＝AI（Claude）と外部ツール（Slack・Notion等）を繋ぐ共通規格**。パソコンのUSB-Cポートのように、異なるツール同士を「共通の規格」で繋ぐ仕組み。MCP = Model Context Protocol = モデル・コンテキスト・プロトコル
- **Skills（手順パッケージ）vs MCP（外部ツール接続）の対比**で明確化
- **MCPは接続設定が複雑に見えるが、実は「認証情報をconfig.jsonに書くだけ」の3ステップで完了**
- **Slack MCPで全チャンネルをクロール→AIが日報を作成→毎日30分の作業が30秒**
- **上司向けチーム活動サマリーの自動生成＝経営層にも響く価値**
- **MCPは複数接続可能**（Slack＋Calendar＋Gmail等を組み合わせる入口）

### 受講者のつまずき
- MCPの接続設定（`~/.claude/mcp.json` や `claude_desktop_config.json`の編集）でつまずく
- Skillsと混同される（「Skillも外部ツール連携じゃないの？」）
- 「接続したのに動かない」環境依存トラブル

### 画面操作方針
スライドは表紙＋MCP概念図（USB-C例示）のみ。画面収録で実演。`https://github.com/modelcontextprotocol/servers` 公式リポジトリのslack実装を使用。接続手順（環境構築→トークン取得→config.json記述→接続テスト）→ Claude Codeプロンプト「Slackの過去7日間の全チャンネルをクロール、日報を生成して」→ 結果表示。サンプルワークスペースは架空チーム（実在組織データは使わない）。

### 位置付け方針
実践・導入ハイブリッド（MCP第1弾）。S3全体を「手順パッケージ化（V1 Skills）→自分の業務をパッケージ化（V2 Skills Creator）→外部ツール連携（V3 MCP第1弾）」と階段状に設計。V3はMCP概念の初登場動画としてS3内での役割を担う。V4以降で「複数MCP組み合わせ」へ接続する布石。

### 参照一覧
- 公式MCPサーバーリポジトリ：`https://github.com/modelcontextprotocol/servers`
- Anthropic公式Skills：`https://github.com/anthropics/skills`
- S3-V1（Skillsとの対比）／S3-V2（作り手体験）
- MCP概念図例：`https://ai.plainenglish.io/what-is-mcp-model-context-protocol-18624a444b51`

### シーンの流れ（11シーン）
まず**毎日30分かけてSlack日報を手作業で書いているチームの課題**を紹介し、「この30分が30秒になったら」という価値提示で冒頭1分を使う。次にMCPを日常例で説明する。USB-CやLightningケーブルのようにメーカーが違う周辺機器も「共通の規格」で接続できるのと同じで、SlackもNotionもGmailもMCPという統一規格でClaudeに繋げば異なるツール同士が協調して動く。MCPの定義を明示し、Skills（手順パッケージ）との違いを対比で明確にしたら、Slack MCPの接続実演へ。公式リポジトリからSlack MCP実装を確認、トークン取得方法（Slack APIコンソール）、config.jsonへの設定記述、接続テストで「接続完了」を確認する3ステップを丁寧に解説する。ここが受講者のつまずきポイントなので「接続できない時はここをチェック」という実務チェックリストを口頭で補足。次に接続済みのSlack MCPに「過去7日間、すべてのチャンネルのメッセージをクロール。営業の活動・企画の進捗・事務作業の完了等を集約して、経営層向けの日報を生成して」というプロンプトで指示し、実際の出力を見せる。構造化された日報（日付・チャンネル別活動サマリー・数字による成果指標・上司向けコメント）を表示。同じPromptで週報も生成可能なことを触れる。HTML化する場合の利点を軽く説明。最後に「今見たのはSlack MCPだけだが、同じやり方でNotionの議事録・Figma・X など複数ツール組み合わせまですべてMCP経由で可能」とV4〜V6への布石を語ってクロージング。

### 設計ノート
- **日常との橋渡し**：USB-Cポート例でMCPの抽象性を緩和
- **対比構造**：Skills vs MCP（混同防止）
- **次セクション布石**：V4〜V6のMCP連続性

## S3-V4 Notion MCP（構成案ドラフト）

### 再定義の核
Notionは単なる「タスク管理ツール」ではなく、組織全体の知識・意思決定・プロジェクト状況が蓄積された**「組織の記憶の器」**。MCPで接続すると、Claude Codeが組織のあらゆる情報に直接アクセスし、横断的に検索・要約・再構成できる力が手に入る。個別の定型タスク処理ではなく、組織全体の知識活用が加速する。

### ゴール
Notion MCPを接続し、組織内に散在する知識（議事録・プロジェクト情報・ドキュメント・ナレッジ）を横断して検索・抽出し、新しい視点からHTML報告書や組織内FAQなどに再構成できるようになる。

### 重要トピック
- Notion MCPの接続方法と権限設定
- **組織の記憶をClaudeが横断操作することの価値**（個別タスク処理→知識の統合活用へシフト）
- 議事録×プロジェクト×ドキュメント×ナレッジの4層構造を一度に検索する力
- **「過去3ヶ月の議事録から経営課題を抽出」**実例
- オンボーディング資料自動生成のユースケース
- 週報・ダッシュボード・FAQ自動化を通じた「情報の民主化」
- **「Notionを開かずにNotionを操作する」から「Notionを読み込んで意思決定を支援する」への進化**

### 受講者のつまずき
- 「Notion MCPって結局タスク登録の自動化?」という限界感（実は知識横断の力学が本質）
- Notion APIの権限設定が複雑（Integration・Database Permission・User権限の3層理解がない）
- 「どんなデータが取れるのか」の実感がない

### 画面操作方針
スライドは表紙のみ。画面収録で実演。デモは**「過去3ヶ月の議事録から経営課題を抽出してHTMLダッシュボード生成」**シナリオ。Notion内の議事録DB（複数プロジェクト分）→ Claude Codeでの横断検索→HTML出力の一気通貫フロー。個人情報（社員名等）はダミーデータで統一。

### 位置付け方針
実践（MCP第2弾）。S3-V3（Slack＝流れる情報・リアルタイム）に対して、V4はNotion＝蓄積された情報・歴史という**対比構造**を作る。「流れる情報」と「蓄積された情報」の両立で、組織の意思決定速度と質が同時に上がる価値を伝える。V5 Figma・V6 X MCPへの布石として「MCP＝組織のあらゆる場所にClaudeが入る」連続性を意識させる。

### 参照一覧
- Notion公式MCP：`https://github.com/makenotion/notion-mcp-server`
- Notion Databasesドキュメント：`https://developers.notion.com/reference/database-query`
- S3-V3（Slack MCP）との対比
- S3-V5 Figmaへの接続地点

### シーンの流れ（11シーン）

## S3-V5 Figma MCP（構成案ドラフト）※マネーフォワードから差し替え

### 題材差し替えの意図
マネーフォワード連携は経理専門の閉じたユースケースであり、講座受講層（経営者・マーケター・人事・デザイナー）への横串の訴求力が限定的。一方、Figma MCPは「デザイン↔実装の橋渡し」という全職種共通の課題に取り組む題材であり、デザイナー・PdM・フロントエンドエンジニアが同時に価値を感じられる。Claude Codeからデザインをそのまま読み取り、コードに落とし、結果をブラウザで確認する一気通貫な流れを通じて、「AIが組織のあらゆる場所に入る」というS3全体の姿勢が最も鮮烈に伝わる。

### Figma MCP調査サマリ

**公式版（Figma Dev Mode MCP Server）**
- フル機能：デザインデータ読取（get_design_context）、変数・スタイル定義抽出（get_variable_defs）、Canvas書き込み（Figmaコンテンツ作成・修正）、FigJam統合、Component/Auto-layout対応
- セットアップ：`claude mcp add --transport http figma https://mcp.figma.com/mcp` またはDesktop AppのDev Modeから有効化
- 対応クライアント：Claude Code、Cursor、Windsurf、VS Code

**コミュニティ版（GLips Figma-Context-MCP）**
- REST APIベース（読み取りのみ）、応答を99.5%圧縮、軽量・シンプル、Canvas書き込みなし、FigJam非対応

**推奨：講座では公式版**。受講生が「AIからの逆方向（Claude→Figma）の操作」も体験でき、「設計↔実装の双方向性」の本質に迫れるため。Anthropic・Figma両社の正式対応で学習の持続性が高い。

### 提案タイトル

### ゴール
Figma MCPを接続し、FigmaのデザインカンプをClaudeが直接読み取り、その場で実装コードを生成し、ブラウザで動作確認できるようになる。同時に、「デザイナーの頭の中にAIが入る」ことで、設計と実装の翻訳コストが劇的に削減できることを体験する。

### 重要トピック
- **Figma MCPとは＝AIが「デザインシステム」を読むトンネル**（日常の「設計図を読む大工」「レシピを読む料理人」と同じ発想で「デザイナーの頭の中にAIが入る」状態を実現）
- **デザイン⇔実装の翻訳コストの削減**：言語化の手間がなくなる（Figmaで「こう見たい」と示すだけ→Claudeが最適な実装を判断）

- **実務レベルのデザインシステムの必要性**：トークン化・コンポーネント体系・Variant設定が実装精度を左右

## S3-V6 X MCP（構成案ドラフト）※X APIから差し替え

### 題材差し替えの意図
現状のS3-V6はX APIの直接呼び出しで、APIキー管理・エラーハンドリング・レート制限対応が学習負荷になっている。一方、xdevplatform/xmcpはX公式が提供するMCPサーバーとして投稿・検索・ユーザー情報取得など119のツールをFastMCP経由で提供。**S3全体が「Skills→Slack→Notion→Figma→X」というMCP統一で通る**ことで、V6も「MCPプラグインを入れるだけでAPIキー管理の手間なくSNS自動化できる」爽快感を体験できる。マーケター向けには「SNS運用時間削減」、個人発信者向けには「毎日の投稿戦略をAIに任せる」という訴求が響き、セクション3の締めくくりとして「AIが組織のあらゆる接点（内部→外部）に入る」ストーリーが成立する。

### X MCP調査サマリ

**機能提供**
- xdevplatform/xmcpはX公式が提供するPythonベースのMCPサーバー（FastMCP使用）
- 提供ツール：createPost（投稿）、searchPostsRecent（最新投稿検索）、getUsersMe（自分のユーザー情報）、getLikes（反応取得）、getRetweetedByMe 等、119個のツール群
- ストリーミング・Webhookは非対応

**認証・セットアップ**
- X Developer Portalでのアプリ作成・APIキー取得が前提
- xmcpサーバー側はPythonパッケージ経由で比較的シンプル（FastMCPで自動化）
- Claude Code側はMCPサーバーとして登録、設定ファイルでxmcpエンドポイントを指定

**料金・レート制限**
- xmcpサーバー自体は無料、X API利用料が主な課金対象（v2エンドポイント従量課金）
- レート制限：X API仕様（例：投稿300件/15分）

**npaka氏解説**：「FastMCPで119個のツール自動生成」「認証情報はxmcpサーバーに保持、Claude Codeからは透過的に利用」
