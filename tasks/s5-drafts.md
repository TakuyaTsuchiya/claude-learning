# S5 改善ドラフト（V1〜V6、計6本）

**前提**：旧S5-V2（セキュリティ）はS2-V6へ移動済み。S5は7本→6本にリナンバ。既存の video-num 重複バグ（V3が2つ）もここで解消。

**適用した共通改善6原則**：
1. Step表記撤廃（→「まず／次に／最後に」の話し言葉）
2. 冒頭50〜60秒の「なぜ重要か」宣言を強化
3. 日常比喩を追加
4. シーン数9〜12に拡充
5. 受講者のつまずき2〜3個に増やす
6. 次V予告で連続性を作る

## S5-V1：GitHub入門 — アカウント作成＆初めてのリポジトリ

### ゴール
GitHubアカウントを作成し、Claude Codeからリポジトリを作成・コミット・pushできるようになる

### 重要トピック
- GitHubとは「コードのクラウド保管庫」
  - 日常比喩：Google Driveのコード版
- コミット＝保存ポイントを作る／プッシュ＝クラウドに送る／プル＝クラウドから持ってくる
- gh auth login（ブラウザ認証）
- Claude Codeからのgit操作（git add / commit / push を自動でやってくれる）
- プライベートリポジトリとパブリックの使い分け

### 受講者のつまずき
- 「Gitは難しい」という先入観（→Claude Codeが操作してくれるので、手元で覚えるコマンドはほぼ不要）
- gh auth loginのブラウザ認証フローで迷う（→画面収録で一気通貫に見せる）
- 何をコミットすればいいか分からない（→「作業の区切りで毎回コミット」をデフォルト化）

### 画面操作方針
スライドは表紙＋コミット／プッシュ／プルの概念図のみ。画面収録で実演：GitHubアカウント作成 → gh auth login → Claude Codeでリポジトリ作成 → 簡単なファイル作成 → コミット＆プッシュ → GitHub画面で反映確認

### 位置付け方針
S5の出発点。**「月額払ってたSaaSを自作する」体験**の第一歩として、まず道具箱を整える。V2（Vercel）・V3（Supabase）もセットアップ回なので、V1〜V3の3本で土台を作り、V4以降で実際のアプリ作りに入る戦略

### 参照一覧
- gh CLI公式: https://cli.github.com/
- GitHub公式（アカウント作成）: https://github.com/signup

### シーンの流れ（9シーン）
まず冒頭で「SansanやSalesforceに月額何万円も払っているツール、実は自分で作れるとしたら？このセクションではそれを体験します。そのための道具箱を整えるのが、まずV1〜V3の3本です」と宣言 → GitHubとは何か（コードのクラウド保管庫、Google Driveのコード版）→ コミット／プッシュ／プルの全体像（概念図）→ GitHubアカウント作成（ブラウザ）→ gh auth loginでClaude Codeと連携 → Claude Codeでリポジトリ作成 → 簡単なテキストファイルを作ってコミット＆プッシュ → GitHub画面で反映確認（「クラウドに保存できた」の体感）→ 最後に「次のV2でVercelを使って、このリポジトリを世界に公開します」と予告

## S5-V2：Vercel入門 — アカウント作成＆初めてのデプロイ

### ゴール
Vercelアカウントを作成し、GitHubリポジトリと連携して、Webページを世界に公開できるようになる

### 重要トピック
- Vercelとは「GitHubのコードを自動でWebサイトにしてくれるサービス」
  - 日常比喩：noteやTumblrに記事を書くと公開URLができるのと同じ感覚
- GitHub連携（リポジトリをpushすると自動デプロイ）
- デプロイの概念：「URLを配れる」＝誰でもアクセスできる
- 無料プラン（個人利用なら十分）
- URLの仕組み（vercel.appサブドメイン／カスタムドメイン）

### 受講者のつまずき
- 「デプロイ」「本番公開」という言葉が重く感じる（→ボタン1つ、無料、いつでも取り消せる）
- 「全世界に公開されるのが怖い」（→URLを教えなければ事実上誰も見ない／プライベート公開も可）
- GitHubとVercelの関係が分からない（→「コード保管庫（GitHub）」と「公開サービス（Vercel）」の役割分担）

### 画面操作方針
スライドは表紙＋GitHub→Vercelの連携図のみ。画面収録で実演：Vercelアカウント作成 → GitHub連携 → V1で作ったリポジトリを選択 → デプロイ → 発行されたURLにアクセス → スマホでも見られることを確認

### 位置付け方針
S5のセットアップ2本目。V1で作った「コードの保管庫」をV2で「**世界に公開する窓口**」に繋ぐ。V4以降の3つのアプリは全てこのVercel経由で公開するので、ここで仕組みを固める。完成品のURLを他人に送れる体験が、作る動機を強くする

### 参照一覧
- Vercel公式: https://vercel.com/
- S5-V1で作ったGitHubリポジトリ（デプロイ対象）

### シーンの流れ（9シーン）
まず冒頭で「V1でGitHubにコードを置きました。でもそれだけでは誰もWebで見られません。コードをWebサイトに変えて世界に公開する仕組みが必要です。それがVercelです」と宣言 → Vercelとは（GitHubとセットで使うサービス）→ 日常比喩（noteに記事を書くと公開URLができるのと同じ）→ Vercelアカウント作成 → GitHub連携 → V1で作ったリポジトリを選ぶ → デプロイ実行 → 発行されたURLにアクセス（「これで世界中からアクセスできる」をスマホでも確認）→ 最後に「これで道具が2つ揃いました。次のV3でデータベース（Supabase）も先にセットアップしておき、V4から実際のアプリ作りに入ります」と予告

## S5-V4：名刺管理アプリを作る（Sansan風）

### ゴール
アプリ開発用CLAUDE.mdを整備し、スペック駆動で名刺管理アプリを実装、GitHubにpushしてVercelで公開するまでの全フローを1本で体験する

### 重要トピック
- **アプリ開発向けCLAUDE.mdの書き方**（S2-V2のCLAUDE.mdとは別、**アプリ開発スケールの書き方**：技術スタック・ディレクトリ構造・コーディング規約）
- **スペック駆動開発のアプリ開発スケール適用**（S2-V5では単一HTMLで体験済／ここでは複数画面のアプリに拡張）
  - 日常比喩：「Sansanに月額払ってるやつ、自分で作れるとしたら？」
- 名刺管理アプリの要件：一覧／詳細／検索／新規登録／編集／削除（CRUD）
- **ローカルストレージ保存**（S5ではまだDB使わない）
  - 日常比喩：自分のブラウザの中だけに残るメモ／冷蔵庫に貼った付箋（その家でしか見えない）
  - 対比：クラウドDB（Supabase）= 端末を超えて共有できる／ローカルストレージ = 自分のブラウザだけ
- V1→V2→V4の一気通貫：コード書く → GitHub push → Vercel自動デプロイの完成形

### 受講者のつまずき
- 「自分でアプリを作れる気がしない」（→スペック駆動＋Claude Codeで、1時間もかからず作れる実例を見せる）
- CLAUDE.mdに何を書けばいいか分からない（→アプリ開発用テンプレを実例で提示：技術スタック・ディレクトリ構造・コーディング規約）
- **「ローカルストレージって何？」**（→自分のブラウザの中だけに残るメモ／他の端末や他人とは共有できない／冷蔵庫の付箋的な位置付け）
- 「Sansanのような本物にはならないのでは」（→機能としてはほぼ同等／違いはデータが自分のブラウザだけに保存されること／S6でクラウドDBに繋げば本物になる）

### 画面操作方針
スライドは表紙＋完成画面キャプチャのみ。画面収録で実演：CLAUDE.md作成 → スペック駆動で要件定義 → Claude Codeに実装依頼 → ローカルで動作確認 → GitHub push → Vercel自動デプロイ → 発行URLでスマホからも確認

### 位置付け方針
S5の**核となる回**。V1〜V3のセットアップが「道具揃え」なら、V4は「初めてのSaaS自作体験」。ここで覚えた「CLAUDE.md → スペック駆動 → 実装 → push → デプロイ」のフローが、V5・V6の前提になる。S2-V5で学んだスペック駆動の考え方を、**アプリ開発スケールに拡張**する回でもある

### 参照一覧
- S2-V2「CLAUDE.mdの書き方」（基礎編）
- S2-V5「スペック駆動開発を実践しよう」（単一HTML版）
- S5-V1（リポジトリ作成）／S5-V2（デプロイ）

### シーンの流れ（10シーン）
まず冒頭で「Sansanのような名刺管理SaaSに月額数千円払っている会社は多いです。でも、Claude Codeとスペック駆動があれば、同等の機能を1時間で作れます。それを今日体験します」と宣言 → アプリ開発用CLAUDE.mdの書き方（技術スタック・ディレクトリ構造・コーディング規約をテンプレで提示）→ S2-V5のスペック駆動をアプリ開発スケールに拡張（複数画面のアプリは、要件を先にまとめないと迷子になる）→ 要件定義（一覧／詳細／検索／新規登録／編集／削除）→ Claude Codeに実装依頼 → ローカルで動作確認 → GitHubにpush → Vercelが自動でデプロイ → 発行URLでスマホからも確認 → 最後に「見た目はSansan、機能もほぼ同等。違いはデータがローカル保存だけ。これはS6でSupabaseに繋いで解決します。次のV5は経費アプリで、同じフローを別ドメインで反復します」と予告

## S5-V6：CRMを作る（Salesforce風）

### ゴール
Salesforce風CRMを実装して、V4・V5で学んだフローを3回目の反復で定着させ、「もう自分で好きなSaaSを作れる」という独り立ち感覚を得る。S5のまとめとして、3つのアプリが揃った達成感とS6への展望で閉じる

### 重要トピック
- Salesforce風CRMの要件：顧客一覧／商談管理／パイプラインステータス（見込み→提案→成約）／フォローアップメモ
- **V4・V5と同じフローを3回目**：CLAUDE.md → スペック駆動 → 実装 → push → デプロイ
- CRM特有の新要素：
  - **パイプラインのステータス遷移**（カード形式で視覚化、ドラッグ操作）
  - **顧客と商談の紐付け**（1対多のデータ構造）
- 日常比喩：「Salesforceに月額何万円も払ってる営業管理、自分で作れる」
- S5総まとめ：3つのSaaS風アプリ完成／ローカル完結の限界とS6での解決予告

- ステータス遷移の実装が難しそう（→「ドラッグで動かせるようにして」で伝えるだけでOK／Claude Codeが適切な仕組みを選ぶ）
- 「3つ作ったけど全部自分のブラウザの中だけ、実務では使えないのでは」（→S6でクラウドDBに繋げばチーム共有の本物になる、という次への期待を作る）

S5の**総まとめ回**。V4（名刺）・V5（経費）・V6（CRM）で3つのSaaS風アプリを作り終えるマイルストーン。3回の反復で「自分一人で好きなSaaSを作れる」独り立ち感覚を確立する。次のS6でバックエンドを繋げば、3つのアプリが**チームで使える実用レベル**に昇格する、という橋渡しで閉じる

- Salesforceの機能参考（任意）

## S6側のクロスリファレンス（既存バグが自動修正される）

**重要な発見**：S6側は既に `S5-V4（名刺）／S5-V5（経費）／S5-V6（CRM）` と参照していたが、これは旧S5のナンバリング（名刺=V5、経費=V6、CRM=V7）とズレていた＝**既存バグ**。
