# Claude Code講座 教育方針・コンセプトメモ

土屋さんの講座に対する考え方・方針をまとめたもの。台本作成・構成レビュー時の判断基準として使う。

---

## 1. 「箱から出したまま」→「自分の道具にする」

- Claude Codeは箱から出したままでも十分使える（Boris Cherny語録）
- S1で「何も設定しなくても魔法が起きる」体験 → S2以降で使いこなすスキル
- 先に完成形を作って、後から仕組みを学ぶ

## 2. 「概念より先に動きを見せる」

- この講座は**動画**であることが最大の差別化
- ネット上のテキスト教材と違い、具体的な操作画面を見せられる
- 概念説明は最小限 → まず動いてるところを見せる → 後から解説
- 例：GitHubは「単純にセーブポイントを作るだけ」と見せれば一発でわかる

## 3. 「ローカル便利ツール量産」で学ぶ

- 土屋さん自身の習得方法：Webにデプロイせず、社内向け便利ツールをローカルで作りまくって覚えた
- セキュリティをそんなに気にしなくていい（ローカル完結）
- 手を動かして先に完成形を作り、後から仕組みを学ぶ

## 4. 社内ツール作りの「横展開」ビジネス価値

- 社内の業務を効率化 → 評価が上がる
- コンサルタント的な動きもできる：
  - 他の会社のボトルネックを見つける
  - どの会社も基本的に同じようなことで悩んでいる
  - 横展開（水平展開）が可能
- ユースケース：営業、採用、バックオフィスなど
- 「御社専用の便利ツールをサクッと作れます」
- 「Claude Codeの有料サブスクリプションだけで達成できます」
- → S3-S4（営業・事務系トラック）の核心メッセージ

## 5. トークン管理とスクリーンショット活用

- トークンが増える → コンテキストが溢れる → AIが迷う
- こまめに/clearすることが重要
- トークン = 文字数。テキスト < 画像 < 動画の順に消費
- **会話が長くなるほどトークン消費は雪だるま式に増加**（10回のやり取りで想定の4倍以上）
- **/compactは推奨しない**（緊急避難的な手段）：
  - 圧縮で細かなニュアンスや文脈が失われ、理解精度が低下する
  - 「さっき言ったはずなのに忘れている」「的外れな回答が増える」現象が起きる
  - /compactが必要になる時点でセッション管理に問題がある
  - 原因1：/clearを適切なタイミングで使っていない
  - 原因2：1セッションで扱うタスクの粒度が大きすぎる
  - 「認証システム全体を作り直して」→「ログイン処理を実装して」「トークンリフレッシュを追加して」と分割する
  - 参考: https://zenn.dev/lv/books/f611cdfe8a6e1a/viewer/695602
- Opusで計画、Sonnetで実装のモデル使い分けも有効
- 不要なMCPツールを入れすぎない（ツール数もトークンを食う）
- スクリーンショット活用：アプリを動かす → スクショ → Claude Codeに見せてフィードバック → 開発加速
- 非エンジニアでも「見せて直してもらう」ができる
- 参考: https://zenn.dev/aun_phonogram/articles/4be2f4745726fb （トークン消費の雪だるま式増加の解説）

## 7. CLAUDE.mdの階層構造を教える

土屋さん自身が初心者時代に不思議に思ったポイント。公式ドキュメントで明確化済み。

**CLAUDE.mdの4つのレイヤー（優先度：下ほど高い）：**

| スコープ | 場所 | 用途 |
|---------|------|------|
| 管理ポリシー | `/Library/Application Support/ClaudeCode/CLAUDE.md`（Mac） | 組織全体のルール（IT管理者用） |
| ユーザー指示 | `~/.claude/CLAUDE.md` | 個人の好み（全プロジェクト共通） |
| プロジェクト指示 | `./CLAUDE.md` or `./.claude/CLAUDE.md` | プロジェクト固有のルール（チーム共有） |
| サブディレクトリ | 各サブフォルダ内のCLAUDE.md | フォルダごとの指示（オンデマンド読込） |

**読み込みタイミング：**
- セッション開始時に読み込まれる（システムプロンプトではなくユーザーメッセージとして）
- /compactの後も再読込される（CLAUDE.mdは圧縮を完全に生き残る）
- サブディレクトリのCLAUDE.mdはオンデマンド（そのフォルダのファイルを読むとき）
- CLAUDE.mdはコンテキストであり強制設定ではない → 具体的で簡潔なほど従う

**自動メモリ（MEMORY.md）：**
- Claudeが自分で書くメモ（ビルドコマンド、デバッグ知見など）
- `~/.claude/projects/<project>/memory/` に保存
- 最初の200行 or 25KBが各セッション開始時に読み込まれる

**効果的なCLAUDE.mdの書き方（公式Best Practicesより）：**
- `/init`でスターターCLAUDE.mdを自動生成 → 時間をかけて改善
- 必須の形式はない。**短く人間が読める状態に保つ**
- すべてのセッションで読み込まれるため、**広く適用されるもののみ**を含める
- ドメイン知識やたまにしか使わないワークフロー → CLAUDE.mdではなく**Skills**に入れる
- 各行に問いかける：「これを削除するとClaudeが間違いを犯すか？」→ Noなら削除
- 膨らんだCLAUDE.md → Claudeが実際の指示を無視するようになる
- CLAUDE.mdをコードのように扱う：定期的にレビュー・削除・テスト

**トラブルシューティング：**
- Claudeがルールを無視する → CLAUDE.mdが長すぎてルールが埋もれている
- Claudeが既に書いてあることを聞いてくる → フレーズが曖昧

**講座での教え方：**
- 「パソコン全体のルール」「このプロジェクトだけのルール」「このフォルダだけのルール」と日常の比喩で説明
- まずプロジェクトCLAUDE.mdだけ教える → 慣れたらユーザーCLAUDE.mdを紹介
- `/init`コマンドで自動生成できることも教える
- 「CLAUDE.mdはダイエットと同じ。太らせすぎると効かなくなる」

**参照：** https://code.claude.com/docs/ja/memory

## 8. Skills = 自社業務の資産化（営業・事務系の核心）

Claude Codeで重要なのは：
1. **コンテキストを管理する**（CLAUDE.md、/clear、トークン管理）
2. **フォルダやファイルを育てていく**（スキル、ドキュメント、設定を蓄積）

**Skillsが特に重要な理由：**
- 一度やったこと、繰り返しやることはSkillに登録すべき
- 世の中のスキルをインストールして使うことよりも、**自社の業務をスキルに登録して社内で横展開する**方がはるかに価値が高い
- スキルクリエイター（skill-creator）のインストール方法を必ず教える

**ビジネス価値：**
- 自分が作ったスキルを部署内で共有 → 「このスキル発動するだけでOK」
- 他の会社にも同じ課題がある → コンサルとして横展開可能
- 業務ノウハウがスキルとして資産化される

**Skillsの育成 = Agent開発の本質（Zenn記事より）：**
- 従来：ドメイン知識は「人の頭の中」か「ドキュメント」→ 聞かないと伝わらない、書いても読まれない
- Skills：ドメイン知識を「Claudeが実際に使える形」で蓄積できる → 単なるドキュメントではなくClaudeの行動を直接規定
- 「うちのエラーハンドリングはこうする」「PRレビューではここを見る」がチームの共有資産に
- Skillsの育て方が企業間の競争力の源泉になる
- **skill-creatorプラグインは必須**
- 参考: https://zenn.dev/lv/books/f611cdfe8a6e1a/viewer/e39efc

**講座での配置：S3（営業・事務系トラック）に入れる**
- S3既存テーマ：カスタムスラッシュコマンド、Agent Skills（SKILL.md）の基礎
- ここに「自社業務をスキル化して横展開」の実践を追加
- skill-creatorのインストールと使い方も含める

## 9. GUIポチポチからの脱却：MCP/CLIの本質

**MCP（Model Context Protocol）とCLI（コマンドラインツール）は似て非なるもの。だが背景にある思想は同じ：**

- GUIを何回もポチポチクリックするのは実はすごく非効率
- 同じことをAPI経由で、CLIやMCPを使って自然言語でClaudeに操作させる
- これが画期的

**具体例：**
- freeeの経費精算：GUI→ログイン→メニュー→入力→確認→送信（5ステップ）
  → MCPなら「今月の交通費を経費精算して」で完了
- WordPress記事投稿：管理画面→新規投稿→カテゴリ選択→公開
  → CLIなら「この原稿をWordPressに投稿して」で完了

**講座での教え方：**
- 「毎日GUIで5分かけてることを、一言で終わらせられる」
- MCPとCLIの違いは軽く触れる程度でOK。重要なのは「自然言語でAPI操作できる」という概念
- S3-S4（営業・事務系トラック）で実演

## 11. mkdir + cd = 「話題ごとにフォルダを分ける」習慣

**絶対覚えてほしいコマンドはmkdirとcdだけ：**
- `mkdir プロジェクト名` → フォルダを作る
- `cd プロジェクト名` → そのフォルダに入る
- この2つだけでClaude Codeの作業環境が整う

**本質：1セッション1話題の原則**
- コンテキストウィンドウ・トークン管理に直結
- 1つのセッションで複数の話題を混ぜない → AIが迷わない
- 話題が変わったら新しいフォルダを作って、そこでClaude Codeを起動する
- フォルダ = プロジェクト = 1つのテーマ

**講座での教え方：**
- S1のターミナル基礎で「mkdirとcdだけ覚えればOK」と明言
- pwd/lsも教えるが、最重要はこの2つ
- 「フォルダを分ける = 話題を分ける = AIが賢く動く」という因果関係を示す

## 13. 音声入力で開発を加速する

- Claude Codeのボイスモード、またはAqua Voice / Typeless（有料ツール推奨）
- 自然言語で会話しながら開発 → スピードが上がる

- 非エンジニアにとっては特に「話しかけるだけでコードが書ける」体験が強烈

## 14. サンドボックス = 箱から出したままで安全

- macOSではSeatbeltでファイル/ネットワークアクセスを制限
- プロジェクトフォルダ外のファイルを勝手に読み書きできない
- 箱から出したまま（out of the box）で動く → 初心者でも安全
- 参照: https://code.claude.com/docs/ja/sandboxing
- Boris語録の「/permissionsで安全に」もこの文脈
- S2-V4「セキュリティの基本とサンドボックス」で教える

## 15. Planモード = 考えてから動く（基本中の基本）

**公式Best Practicesより：**
> "最初に探索し、次に計画し、その後コーディングする。研究と計画を実装から分離して、間違った問題を解決することを避けます。"
> "Claude が直接コーディングにジャンプさせると、間違った問題を解決するコードが生成される可能性があります。"

- Shift+Tab×2でPlanモードに切り替え
- **探索 → 計画 → 実装** の3ステップ
- いきなりコードを書かせない = Vibe Codingからの脱却
- 初心者がやりがちな「全部おまかせ」を防ぐ
- Boris語録でもPlan mode活用を推奨

**講座での配置：**
- S2の基本コマンドの回（V1）で「Planモードで作る習慣」を教える
- S5のスペック駆動開発でさらに深掘り
- 「考えてから動く」はプログラミングに限らず業務全般に通じる

## 18. 裏テーマ：AI DXコンサルになれるレベルへ

- 表テーマ：Claude Codeを使えるようになる
- **裏テーマ：AI DXコンサルタントとして動けるレベルまで持っていく**
- もちろん即戦力には経験が必要。だが「型」を身につければ成長速度が段違い
- そのために**社内便利ツールをたくさん作る** = 数学でいう「青チャート」
  - 青チャート = 典型問題のパターンを網羅的に解いて、どんな問題にも対応できる力をつける
  - 社内ツール量産 = 典型的な業務課題のパターンを網羅的に解決して、どんな会社にも対応できる力をつける
- 講座で扱うユースケース（S3-S4）は「問題集」の役割。受講後に自分で量産する習慣が本番

## 19. 検証ループを与える = 最もレバレッジが高い行為

**公式Best Practicesより：**

> "Claude に自分の作業を検証する方法を与える。テスト、スクリーンショット、または期待される出力を含めて、Claude が自分自身をチェックできるようにします。これはあなたができる最も高いレバレッジのことです。"

- Claudeはテスト実行・スクショ比較・出力検証など「自分で確認」できると**劇的に良くなる**
- 明確な成功基準がないと、正しく見えるが実際には機能しないものを生成してしまう
- 検証ループがないと、あなたが唯一のフィードバックループになり、すべての間違いがあなたの注意を必要とする
- Boris語録でも「検証フィードバックで品質が2-3倍向上する」

**講座での教え方：**
- S1の魔法体験で「スクショ撮ってClaude Codeに見せる」を実演 → これが検証ループの入門
- S2で「テストを書かせる」「期待される出力を伝える」習慣を紹介
- 「AIに丸投げするな、検証方法を一緒に渡せ」がメッセージ

## 20. 早期かつ頻繁に方向転換する

**公式Best Practicesより：**
> "最良の結果は、タイトなフィードバックループから来ます。"
> "1つのセッションで同じ問題についてClaudeを2回以上修正した場合、コンテキストは失敗したアプローチで乱雑です。/clearを実行し、学んだことを組み込んだより具体的なプロンプトで新しく開始します。"
> "より良いプロンプトを持つクリーンなセッションは、ほぼ常に蓄積された修正を持つ長いセッションを上回ります。"

**操作方法：**
- Esc → Claudeの中途半端なアクションを停止（コンテキストは保持）
- Esc×2 or /rewind → 巻き戻しメニューで以前の状態に復元
- "Undo that" → Claudeに変更を元に戻させる
- /clear → コンテキストをリセットして新しく開始

**講座での教え方：**
- 「2回修正してもダメなら、/clearして最初からやり直す方が早い」
- 失敗したやり取りが残ったセッション ＜ きれいなプロンプトの新セッション
- 「粘るな、やり直せ」がClaude Code流

## 21. 一般的な失敗パターン5つ（公式Best Practicesより）

受講生が必ずハマるパターン。先に教えておくと時間を節約できる。

| 失敗パターン | 症状 | 対策 |
|-------------|------|------|
| **キッチンシンク** | 1セッションで無関係なタスクを混ぜる | タスク間で/clear |
| **何度も修正** | 2回以上修正してもダメ | /clearして学びを込めた新プロンプトで再スタート |
| **CLAUDE.md肥大化** | 重要なルールがノイズに埋もれる | 容赦なく削除。Claudeが指示なしで正しくやることは消す |
| **信頼して検証しない** | もっともらしいがエッジケース未対応 | 常にテスト・スクショ・検証を提供。検証できないなら出荷しない |
| **無限探索** | 「調査して」でコンテキスト爆発 | スコープを狭くするか、サブエージェントに任せる |

**講座での教え方：**
- S2でまとめて紹介「こういう失敗をしがちです。先に知っておきましょう」
- 各パターンは既に他のポイント（#5,#7,#19,#20）で対策を教えてるので、ここは振り返りとして

## 22. プロンプトで具体的なコンテキストを提供する

**公式Best Practicesより：**
> "指示がより正確であるほど、必要な修正が少なくなります。Claude は意図を推測できますが、あなたの心を読むことはできません。"

- 特定のファイルを参照する
- 制約を述べる
- 例のパターンを指摘する
- S1の魔法体験では「雑でOK」と教えるが、S2以降で「具体的に指示するほど精度が上がる」とステップアップ
- S1=雑でも動く体験 → S2=具体的に指示する習慣 → S5=スペック駆動で設計書を渡す、という成長曲線
