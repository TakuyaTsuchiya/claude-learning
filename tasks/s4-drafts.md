## S4-V1 ホテル清掃LMS（構成案・確定版）

### 提案タイトル
「【人事向け】社内eラーニングシステムを自作しよう」（24字）

### ゴール
人事・研修担当者が**現場職向け社内eラーニングシステムのガワ（階層メニュー＋受講ダッシュボード）をClaude Codeで10分で組み上げられる**ようになる。架空「シーサイドホテル」の清掃スタッフ育成LMS（3カテゴリ×3教材＝全9本）を題材に、最低限のスペック駆動開発で仕様書を書き、HTML一枚の受講者ポータルと受講状況マトリクスを動く状態で受講者が手元で再現できるアウトプットを得る。

### 重要トピック
- **現場職の新人教育課題**（日常の「マニュアルを渡しても読まれない／人によって教え方が違う」と同じ構図。外国人スタッフ増・高い離職率・現場OJT負担増という具体状況を、特定チェーンを名指しせず業界全体の傾向として示す）
- **ドメイン知識のプロダクト化**（人事担当者が持つ「清掃手順の分類」「誰がどこまで終わったかの把握」という暗黙知を、S1-V2で学んだHTML単品プロダクト化の発想でそのまま形にする）
- **最低限のスペック駆動開発**（S2-V5で扱った仕様書→実装の流れを最短サイズで適用する例。数行の `spec.md` でも「階層メニュー＋ダッシュボード」レベルなら十分に機能する、という中立的な使い分け視点）
- **階層メニューというUIパターン**（日常の「本の目次」「動画サイトのカテゴリ」と同じ発想。3カテゴリ→各3教材のドリルダウンで、9本でも迷わない導線になる）
- **受講ダッシュボード＝受講者×教材マトリクス**（日常の「学校の出席簿」と同じ構造。田中さん・山田さん・グエンさんの3名×9教材の一覧で「誰がどこで止まっているか」が一目で分かる）
- **スコープを絞る設計判断**（初回は「ガワだけ」。動画再生・ログイン認証・進捗の永続化は載せず、ダミーの進捗フラグをHTML内にハードコードする。この割り切りが10分構築を成立させている）
- **仕様書なしVibe Coding vs 仕様書あり最低限スペック駆動の差**（同じ「LMSのガワ」でも、仕様書があると階層・画面数・ダミーデータが狙い通りに出やすくなる。Vibeが悪いのではなく、コンテキスト量の差）
- **月額LMS vs 自作HTML**（両者は優劣ではなく「多拠点・多ユーザー運用なら月額LMS、自社要件の試作・小規模運用なら自作HTML」と用途で使い分ける）
- **S5への布石**（永続化・複数拠点展開・アカウント管理まで進めたくなった段階で、DB・バックエンドが必要になる。そこはS5で扱う）

### 受講者のつまずき
1. **「eラーニングシステムは専用SaaSを契約しないと作れない」という思い込み**（自分の業務でLMS導入検討に触れた人ほど、ゼロから作る発想が浮かばない）
2. **「仕様書に何を書けばいいか分からない」**（スペック駆動と聞くと詳細設計書を連想して手が止まる。画面数・データ構造・ダミー値の3点に絞れば数行で済むことを実演で見せる）
3. **「9教材分のHTMLを1枚ずつ手で書くのは現実には不可能」と感じて着手できない**（Claude Codeが階層構造ごとまとめて生成する体験で解消する）

### 画面操作方針
スライドは表紙と冒頭の課題設定1枚のみ。残りは画面収録で実演する。流れは、人事・研修の現場課題スライド→`mkdir demos/hotel-cleaning-lms && cd demos/hotel-cleaning-lms` で作業ディレクトリ作成→`spec.md` をClaude Codeと対話で書き起こし（階層：3カテゴリ×3教材＝9本／画面：トップ／カテゴリ／教材詳細／ダッシュボード／ダミーデータ：田中さん・山田さん・グエンさんの受講フラグをハードコード）→Claude Codeに実装を依頼→生成された `index.html` をブラウザで開く→トップ画面（3カテゴリのカード）→カテゴリ画面（ベッドメイキング配下の3教材カード＋YouTube風サムネ）→教材詳細画面（サムネ＋「受講済みにする」ダミーボタン）→ダッシュボード画面（3名×9教材マトリクス、受講済み／未受講の配色）→最後に「動画再生・ログイン・永続化は入れていない」と明示。個人情報配慮として、実在ホテルチェーン名・実在スタッフ名は一切出さず、架空「シーサイドホテル」と田中さん・山田さん・グエンさんで統一する。サムネ画像はフリー素材または無地プレースホルダーを使う。

### 位置付け方針
S4「業務効率化の実践」の1本目として、**S1「ドメイン知識のプロダクト化」とS2「スペック駆動開発」の合流地点**に置く。S1-V2で受講者は「HTMLレポートは単品で作れる」という器を手にしており、S2-V5で「仕様書を書いてから作ると狙い通りに出る」手順を学んでいる。S4-V1はその2つを人事ドメインで実証し、「自分の職種でもゼロから仕組みを生み出せる」という自信を付けさせるハブ回。S4の残り5本（V2バックオフィス・V3人事・V4経営・V5マーケ・V6営業）に受講者を送り出す起点として、最初に「ガワだけでも動けば価値がある」という割り切り感覚を共有する。同時にS5「アプリ開発」への橋渡しとして、**ガワの限界（複数拠点・多ユーザー・進捗永続化になるとHTML単品では足りない）**を最後に正直に述べ、S5でバックエンド・DB・デプロイが登場する必然性を予告する。

まず**ホテル業界の新人教育課題を腰を据えて宣言する**。外国人スタッフが年々増え、現場OJTの負担が先輩清掃員に集中し、離職率の高さと相まって「教えた人が辞めて、また一から教え直し」が繰り返されている現状を50〜60秒で語り、人事・研修担当者が抱える「マニュアルを渡しても読まれない／人によって教え方が違う」という日常の詰まりと接続する → 次に**本日のアウトプット像を先に見せる**。完成した `index.html` をブラウザで開き、トップの3カテゴリカード→ベッドメイキング配下の3教材→教材詳細→3名×9教材のダッシュボードまで30秒で早送り試写し、「このガワを10分で作る」と宣言する → 題材の整理として**コンテンツ構造を図解する**。3カテゴリ（ベッドメイキング／水回り／最終点検）×各3教材で全9本、受講者は田中さん・山田さん・グエンさんの架空3名という前提を共有し、実在ホテル・実名は一切使わない方針を明示する → **最低限のスペック駆動開発の考え方を1分で導入する**。S2-V5の復習として「仕様書＝AIへの設計指示書」を呼び戻し、今回は階層・画面数・ダミーデータの3点だけ書けば十分、というミニマム運用であることを中立に説明する → `mkdir demos/hotel-cleaning-lms && cd demos/hotel-cleaning-lms` で作業ディレクトリを用意し、**Claude Codeと対話しながら `spec.md` を書き起こす**。3カテゴリ×3教材の階層、トップ／カテゴリ／詳細／ダッシュボードの4画面、3名×9教材のダミー進捗フラグをハードコードする方針、動画再生・認証・永続化は載せないというスコープを明記する → 仕上がった `spec.md` をClaude Codeに渡し実装を依頼する。生成された `index.html` をブラウザで開き、**トップ画面の3カテゴリカードとカテゴリ画面の教材カード＋YouTube風サムネ**を操作しながら、階層メニューが仕様書どおりに出ていることを確認する → **受講ダッシュボードを開く**。田中さん・山田さん・グエンさんの3名×9教材マトリクスで、受講済みは塗りつぶし、未受講は空欄という配色を見せ、「誰がどこで止まっているかが1画面で分かる」という人事視点の価値を短く添える → **スコープの割り切りを正直に述べる**。動画再生はサムネのみ、ログイン認証はなし、進捗フラグはHTMLにハードコード、という3点を明示し、「ガワだけでもこの情報密度なら十分に現場で会話が回る」という設計判断を言語化する → **仕様書なしVibe Codingとの対比を短く挟む**。仕様書なしで「ホテル清掃のeラーニング作って」とだけ依頼すると、階層もダミーデータも狙い通りが出にくくなる、という差分を10秒で中立に触れる → クロージングで、**今日のガワをベースに自社の教材タイトルと受講者名を入れ替えるだけで自部署に持ち込める**ことを強調しつつ、多拠点展開や複数人同時更新まで進めたくなった段階ではバックエンドとDBが必要になり、それはS5で扱う、と予告して締める。

## S4-V2 申請書類チェック校閲（構成案・確定版）

### 提案タイトル
「【人事・経理向け】申請書類チェック校閲を自動化しよう」（26字）

### ゴール
経理・人事・総務のバックオフィス担当者が、申請書Excel10件＋添付PDF・画像（領収書・定期券写真・母子手帳コピー）の混在ファイルを、Claude Codeで対話的に突合させ、「承認可」「要確認」を仕分けたHTMLチェックレポートを10分で受け取れるようになる。月100件超の目視突合500分を、AI一次チェック＋人間最終判定のハイブリッド運用で30分に圧縮する感覚を体得することが学習目的である。

### 重要トピック
- **バックオフィス横断の「申請処理」は職種を越えた共通苦労**（経理の経費精算、人事の育児休業、総務の通勤手当。日常の「郵便物の仕分け」「健康診断の書類チェック」と同じで、「正しいか・不備があるか」の一次判定を延々と繰り返す作業）
- **AI一次チェック＋人間最終判定のハイブリッドが現実解**（100%自動化は幻想。80%を自動仕分けして、残り20%を人間が判断する。日常の「迷惑メールフィルタ」と同じで、AIは除外・フラグ立ての役目、最終承認は人が押す）
- **読み取り精度は素材で変わる、という事実を正直に扱う**（印字PDFほぼ100%、スキャン画像90〜95%、手書き・FAX60〜80%。限界を「欠点」でなく「自動で要確認に分類する判断基準」として逆手に取る）
- **個人情報ガードレールが入口条件**（Anthropicデータポリシーの1枚理解＋マスキング実装30秒デモ＋マイナンバー設計除外の思想。技術より先にガードレールを明示する）
- **申請データの現実的な混在**（Excel1ファイル＋領収書画像＋PDF＋手書き写真。ここまでぐちゃぐちゃでもClaude Codeはそのまま読みに行ける、という対話ツールの強み）
- **具体的な差分指摘で信頼を得る**（「申請¥12,300 vs 領収書¥12,500、200円差」「出産予定日 申請5/15 vs 母子手帳5/12、3日差」「経路不一致」のように数値と根拠を並べて出力させる）
- **S3-V2（Skills Creator月次レポート）とバックオフィスの2大柱**（月次集約×申請処理でバックオフィス業務をAIで挟み撃ちにする構図）
- **対話ツール完結 vs Webアプリ化の境界**（S4-V2はサブスク完結、ファイル投入→レポート出力で月100件処理可能。ワークフロー承認・監査ログ・複数ユーザ運用が要るならS5-V6のfreee風Webアプリへ）

### 受講者のつまずき
1. **「個人情報をClaudeに読ませて大丈夫か」**が技術以前のハードル。氏名・生年月日・口座・母子手帳画像をそのまま放り込んでよいのか、社内にどう説明すればよいのかが最大の心理的ブロックになる
2. **「10件ぐらい目で見た方が早いのでは」**という比較の現実感。100件超・月次繰り返しというスケールを提示しないと価値が腹落ちしない
3. **「手書き・スキャン画像の読み取りが不安定だと結局使えない」**の不信感。印字PDFで成功しても、FAX由来の領収書で失敗した瞬間に「やっぱりダメ」となりやすい

### 画面操作方針
スライドは表紙、個人情報配慮、クロージングの3枚のみ。画面収録で実演。
- **冒頭の個人情報シーンを実演前に1〜1.5分で必ず組み込む**：
  1. Anthropicデータポリシー1枚図解（学習非利用・一定期間で削除）を口頭で30秒
  2. マスキング実装デモ30秒（ターミナルで`mask.py`を走らせ、氏名→A氏、生年月日→1980年代、口座→下4桁のみに変換された申請Excelをその場で確認）
  3. マイナンバーは最初から処理対象外として設計した、という思想を一言
  4. 受講者向けガードレール4項目（利用目的通知・マスキング・機微情報除外・社内ポリシー整合）を1枚スライドで明示
- 題材ファイル群を提示：`demos/shinsei-check/` 配下に申請書Excel10件（経費精算5・通勤手当3・育児休業2）＋添付PDF/画像。田中太郎・山田花子・グエン・ティ・ハン等、V1と統一の架空データ
- Claude Codeで対話的に「このフォルダの申請10件と添付を突合し、不整合があれば差額・差日数・経路不一致を具体的に挙げてHTMLレポートにまとめてください」と依頼
- レポート生成中、Claude Codeがファイルを順に読み、画像から金額や日付を抽出する過程を画面に映す
- 生成されたHTMLをブラウザで開き、✅承認可6件／⚠️要確認4件の仕分け、各行の差分根拠（「¥12,300 vs ¥12,500、200円差」等）を確認
- 手書きの不鮮明な領収書が自動で⚠️要確認に落ちる挙動を見せ、「限界が運用ガードレールに転化する」設計を示す

### 重要トピック
- **複合MCP＝「複数の有能な秘書がチームで動く状態」**。個別MCPは1人の秘書が1つの情報源を整理する状態、複合MCPは複数人の秘書が情報源をそれぞれ担当して同じレポートを仕上げる状態として中立に整理（優劣ではなく、扱う情報の広さの違いで説明）。例：Slack担当秘書が顧客の声と社内論議を抽出、Notion担当秘書が経営会議の決定事項を拾い、CEO向け1枚に統合
- **経営層の「月曜朝問題」＝先週の全体像が散らばっている状態**。Slackをスクロールし、Notion議事録を探し、頭の中で時系列に並べ直す作業を毎週繰り返すのは現実には不可能に近い。複合MCPで「**有能な秘書が先週の全体像を月曜朝に用意してくれる**」状態を作るのが本動画の価値（経営層への訴求軸）
- **現場への価値＝週次報告資料作成の繰り返しから解放される**。営業リーダーが金曜夕方にSlackログをまとめて資料化する作業、プロダクトマネージャーが進捗をNotionに書き直す作業が、複合MCPで自動生成される状態に移行。中間管理層・メンバーが「**週次報告資料作成の繰り返しから解放される**」価値も同時に実現する（経営者と現場の両方が得する構図）
- **Slack MCPとNotion MCPの同時起動＝`/mcp` で2つのサーバーがどちらも緑色になっている状態**。家電のUSB-Cハブで複数機器を同時接続するのと同じ発想で、LLMがツールを必要に応じて自律的に呼び分ける
- **4層構造HTML＝経営判断の読み順に情報を並べた設計**。顧客対応動向（外部の声）→ 社内進捗（対応中の施策）→ 経営方針（判断された方向）→ 示唆（次週の焦点）の順で並べることで、CEOが5分で「先週何が起きて、今週どこに注力するか」を掴める
- **チャネル設計が情報密度を決める**。#general、#sales、#cs-support、#sales-voice、#product-dev、#board-updates の6チャネルを横断し、#cs-support と #sales-voice から「顧客の声」を拾うことで、Slack単体でも顧客対応動向を確保できる設計
- **S3-V2との棲み分け＝月次 vs 週次、Skills vs 複合MCP**。S3-V2は固定フローの月次集約をSkillsで資産化する設計、S4-V4は動的な週次リアルタイム集約を複合MCPで走らせる設計。どちらが優れているかではなく、**扱う時間軸と情報の動きに応じた使い分け**で整理する
- **S5への布石＝複数入力を束ねて出力する思考パターン**。複合MCPで「複数ソースを同時に扱い、LLMが自律的にルーティング」を体験しておくと、S5のアプリ開発で出てくるDB設計・非同期処理の思考基盤になる

### 受講者のつまずき
1. **「複合プロンプトの書き方がわからず指示が散らかる」**：個別MCPなら「Slackの先週分を要約して」で済むが、複合MCPでは「SlackのどのチャネルからNotionのどのDBと組み合わせて、どんな順で並べるか」まで指示が必要になり、プロンプトが長く散らかる。→本動画で「4層構造テンプレート」を提示し、パラメータ差し替えで流用できる形に整える

## S4-V6 決裁者アタックリスト（構成案・確定版）

### 提案タイトル
「【営業向け】Webスクレイピングで決裁者アタックリスト自動生成しよう」（34字）

### ゴール
BtoB営業・インサイドセールス・営業DX推進担当が、Sansan事例ページ（`https://jp.sansan.com/case/`）の公開情報を題材に、robots.txt確認→一覧ページのクロール→個別事例ページからの決裁者情報抽出→CSV出力までの多段階ワークフローをClaude Code1本で回せるようになる。手作業で100社分の決裁者アタックリスト（会社名・部署・役職・氏名・業界）を作ると8時間かかる「読む・写す・転記する」の単純労働を数分に圧縮し、空いた時間を営業戦略・顧客関係構築という高付加価値業務に振り向ける感覚を体得することが学習目的である。

### 重要トピック
- **BtoB営業の「決裁者アタックリスト作成」は新規営業の生命線**（稟議・契約の承認権限を持つ決裁者に直接届かないと商談が進まない。日常の「飛び込み先の表札を読む」「名刺交換会で役職を聞いて回る」と同じで、会社名・部署・役職・氏名を1件ずつ拾う作業を100社分繰り返すと8時間が溶ける）
- **「決裁者」と「決済者」は別物**（営業業界標準用語は「決裁者」＝稟議・契約の承認権限を持つ人。「決済者」は代金支払い者の意味になるので誤用。Sansan・ベルフェイス等SaaS各社も「決裁者」で統一している）
- **robots.txtと利用規約の確認は営業職の法令リテラシー**（スクレイピングに着手する前にサイト側の許諾範囲を読む責任。`curl https://jp.sansan.com/robots.txt`で`/case/`が許可されていることを目で確認する習慣。個人情報保護法・特定商取引法の軽い橋渡しを添える）
- **2段階スクレイピングの階層構造**（一覧ページのクロール→個別事例ページの抽出という2階建て。Claude Codeが企業ロゴ・リンクからURLパターン`case`, `cases`, `customers`を自律的に見つけて個別ページへ降りていく様子が多段階ワークフローの核心）
- **人間のドメイン知識がAI精度を上げる**（URLパターン候補`case`, `cases`, `customers`をプロンプトに埋め込むとLLMの抽出精度が跳ね上がる。S1から一貫した「AIに任せきりではなく、人間の業務知識を差し込むとAIが伸びる」というメッセージの再現）
- **手作業100社8時間 vs Claude Code数分の対比**（時間差の正体は「人間が1社ずつ読む作業の自動化」。手作業が悪いのではなく、人間の時間を営業戦略・顧客関係構築という高付加価値業務に振り向け直す発想）
- **生成CSVの現物検証**（`attack_list.csv`をExcelで開き、Sansan公式事例ページ3〜4件をブラウザで突き合わせ、抽出された会社名・部署・役職・氏名が公開情報と一致していることを目で確かめる）
- **S4セクションの締めくくり→S5への質的転換**（V1〜V5は単機能の効率化、V6は「取得→加工→出力」の多段階ワークフロー。今回はCSVをローカルで開くツールだが、S5では同じリストをチームで共有しDBに永続化するSaaS化へ質的に変わる）

### 受講者のつまずき
1. **「スクレイピングって法的に大丈夫なのか」**が営業職にとって最初のハードル。技術以前に「怒られないか」「社内の法務・情シスに説明できるか」が気になって着手できない。robots.txt・利用規約・公開情報の3点を動画の入口で明示しないと最後まで腰が引ける
2. **「100社分のURLを1つずつ貼るのか」**と勘違いして、一覧ページ1本渡すだけで個別ページまで降りていくという2段階の設計が腹落ちしにくい。一覧→個別の階層をClaude Codeが自律的に辿る挙動を目で見るまでピンと来ない
3. **「LLMは毎回結果が揺れるから仕事には使えないのでは」**という精度不信。URLパターン`case`, `cases`, `customers`をプロンプトに書き足した瞬間に抽出結果が安定する体験がないと、プロンプトに業務知識を差し込む意味が伝わらない

まず**BtoB営業の「決裁者アタックリスト作成地獄」を宣言**する。新規営業の商談は決裁者に直接届かないと進まない、だから会社名・部署・役職・氏名を1社ずつ拾う必要がある、100社分を手作業で集めると読む・写す・転記するだけで8時間が溶ける、という現場の実感を50〜60秒で腰を据えて語る。そのうえで「Sansan事例ページを題材に、この8時間を数分に圧縮し、空いた時間を営業戦略・顧客関係構築に振り向け直す」という価値を宣言する → 次に**「決裁者」という用語を中立化して定義する**。「決裁者（けっさいしゃ）」は稟議・契約の承認権限を持つ人、Sansan・ベルフェイス他のSaaS各社が営業業界標準として使う表記であり、「決済者」は代金支払い者の意味になるので誤用、という一次情報を短く挟む → **法令配慮シーンを1〜1.5分でしっかり組み込む**。`curl https://jp.sansan.com/robots.txt | grep -i case`をターミナルで走らせて`/case/`が許可されていることを目で確認し、利用規約・公開情報・スクレイピング許可確認済の3点を口頭で整理し、個人情報保護法・特定商取引法の軽い橋渡しを添える。営業職の法令リテラシーとして「着手する前にrobots.txtと利用規約を読む」という習慣を意義ごと位置付ける → 題材ページ`https://jp.sansan.com/case/`をブラウザで開き、事例一覧の企業ロゴ・会社名・業界が並ぶ構造を見せ、「この一覧からクロールを始める」と見通しを共有する → Claude Codeを起動し、一覧URLを1つ渡して「個別事例ページをクロールし、会社名・部署・役職・氏名・業界をJSONで抽出、最後に`attack_list.csv`にまとめてください」とまず素のプロンプトで依頼し、LLMが企業ロゴ・リンクからURLパターンを探る様子を対話ログで見せる → **ドメイン知識をプロンプトに差し込む体験**に入る。素のプロンプトでは抽出精度が揺れる場面を短く示したうえで、「URLパターン候補は`case`, `cases`, `customers`です」という1行を足した途端に個別ページへの到達精度が安定する変化を対比で見せる。「人間のドメイン知識がAIの精度を上げる」というS1からの一貫メッセージをここで再現する → Claude Codeが一覧→個別ページの2段階を自律的に辿り、会社名・部署・役職・氏名・業界を1件ずつJSONで抽出し、最後にCSVにまとめる過程を対話ログで流す。**2段階スクレイピングの階層構造**が多段階ワークフローの核心であることを言葉で一言締める → 生成された`attack_list.csv`をExcelで開き5列が揃っていることを確認し、さらにSansan公式事例ページ3〜4件をブラウザの別タブで開いて抽出データが公開情報と一致することを視覚的に突き合わせる。「抽出が合っている」を目で確かめる現物検証の一手間を省かない → **手作業100社8時間 vs Claude Code数分の対比**を数字で締める。時間差の正体は「人間が1社ずつ読む作業の自動化」であり、手作業が劣っているのではなく、人間の時間を営業戦略・顧客関係構築という高付加価値業務に振り向け直す発想が本質であることを中立化して伝える。営業目的で利用する前に対象者のプライバシー設定・公開許諾範囲を再確認する責任が営業側にある、という1文もここに添える → **S4セクション全体を俯瞰する**。V1〜V5は単機能の効率化、V6は「取得→加工→出力」の多段階ワークフロー、S4は小さな局所最適化から事業全体を視野に入れた戦略的自動化への卒業だった、とセクションを総括する → クロージングで**S5（SaaS化）への質的転換**を予告する。今回はCSVをローカルで開く個人ツール、S5では同じ決裁者アタックリストを複数人で共有しDBに永続化してWebインターフェースからリアルタイム更新するSaaSへ質的に変わる。「今はCSVをローカル、次はチームで共有する段階」と境界を引き、S4-V6が蝶番であることを明示して動画を閉じる
