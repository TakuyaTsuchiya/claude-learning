## 位置付けの前提（講座側の意図）

- **S6-V1〜V3**：受講者が3アプリ（名刺／経費／CRM）にSupabaseでDB接続してプロトタイプ完成

## 先方に確認したいこと（テーマ別）

### A. 協力範囲の大枠
- [ ] 2本分の動画制作にあたって、BACKSTAGE社として**何をどこまで協力可能か**（例：オフィス撮影／インタビュー／画面提供／事例紹介のみ）
- [ ] 公開可能な会社情報の範囲（会社名・組織規模・役員名・プロダクト名等）
- [ ] NDA／出演同意書等、書面で必要な取り決めの有無
- [ ] 出演候補者（エンジニア何名／役職／実名可否）
- [ ] 撮影・取材可能な時期と所要時間（撮影日候補の目安）
- [ ] 講座リリース後の追加撮影・修正対応の可否

### B. コンテンツ素材・画面の公開可否
- [ ] 社内Slack・Notion・GitHub・CI/CD画面など**ツール画面の撮影可否**
- [ ] マスキング・ぼかし処理が必要な領域（顧客名・個人名・認証情報・社内固有情報）
- [ ] 実リポジトリのソースコード／PR／Issueの映像利用可否
- [ ] 監視ダッシュボード・本番運用指標の画面利用可否
- [ ] 認証・脆弱性対策・セキュリティ実装など**機密性の高い画面**の取り扱い方針

### C. S6-V4「日常開発フロー」の題材候補
- [ ] BACKSTAGE社エンジニアの**日常的なClaude Code活用シーン**として映せそうなもの（例：朝会／コードレビュー／Spec起票／ペアプロ）
- [ ] 「プロはこう使う」と見せたい特徴的な運用習慣
- [ ] 受講者が真似できるレベルの身近なユースケース
- [ ] 出演者の役割・人数（1人インタビュー／複数人対話／ナレーション上から）

### D. S6-V5「本番運用とセキュリティ」の題材候補
- [ ] 公開可能な本番運用事例（アーキテクチャ図／トラフィック規模／可用性指標など）
- [ ] 認証・認可の実装方針（どこまで見せられるか）
- [ ] 本番環境での脆弱性対策・監視・アラート対応の一端
- [ ] CI/CDパイプラインの運用例（成功・失敗・ロールバック含む）
- [ ] スケーラビリティ・パフォーマンス対策（具体事例 or 一般論どちらが可能か）
- [ ] 「プロトタイプと本番の境界」を語れる人の推薦

### E. 撮影・編集スタイル
- [ ] インタビュー形式が望ましいか／ドキュメンタリー風／画面収録中心のどれが現実的か
- [ ] オフィス映像の利用可否（物理撮影／リモート画面のみ／ストック素材で代替）
- [ ] BGM・テロップ・社名表示のガイドラインの有無
- [ ] 先方の広報／法務レビューのタイミングと所要日数

### F. 受講者への接続
- [ ] 動画末尾で視聴者を**BACKSTAGE社側の窓口（採用／問い合わせ／サービス紹介等）**に案内してよいか／案内するならどのURLか
- [ ] 推薦書籍・外部リソースの推薦協力可否
