Claude Code・Cursorをインストールしよう
構成
ゴール：Claude CodeとCursorをインストールし、mkdir→cd→claude起動＋ステータスライン設定ができる
重要トピック：Claude Codeはネイティブインストーラーでインストール（npm非推奨）／MacとWindowsでClaude CodeとCursorをそれぞれインストール／WindowsはGit for Windows（Git Bash）のインストールが前提条件として必須／PATHを通す作業を丁寧に／Cursor無料プラン／Claude Code Proプラン契約／mkdir→cd→claude起動／ステータスライン設定（コンテキスト使用率・レート制限・gitブランチをリアルタイム表示）
受講者のつまずき：PATHが通らない（みんなここで詰まる）／WindowsユーザーがGit for Windowsを先に入れずにclaudeコマンドが動かない／ターミナル操作が初めてでmkdir・cdの意味がわからない
画面操作方針：スライドは表紙のみ。画面収録で実演。Mac・Windowsそれぞれのインストール手順＋/statuslineコマンドでステータスライン設定
位置付け方針：前回までに魔法体験を見せた流れを受けて、ここで自分のPCでClaude Codeが動く状態まで持っていく回。これ以降の全動画はこの回で整えた環境の上で進むため、受講者が挫折せず完走するための土台。環境構築で詰まる人が多いポイント（PATH／Git for Windows）を丁寧にケアする
参照一覧：Claude Code公式セットアップ https://docs.claude.com/en/docs/claude-code/setup／Git for Windows https://gitforwindows.org/（Windows前提条件）／Cursor公式 https://cursor.com/／Claude Code Proプラン https://www.claude.com/pricing／ステータスライン公式 https://code.claude.com/docs/ja/statusline
シーンの流れ：まず冒頭で「環境構築は多くの方が挫折するポイントです。ここで丁寧にやっておけば、この先の全動画が自分のPCで再現できるようになります」と宣言（50〜60秒）→ 次にClaude Codeのネイティブインストーラーをダウンロード → Windowsユーザーは前提条件としてGit for Windows（Git Bash）を先にインストールする点を強調して実演 → PATHを通す手順をMac・Windows両方で実演 → Cursorをインストール（Mac・Windowsそれぞれ）→ Claude Code Proプランを契約 → mkdir→cd→claude起動の流れを実演 → 最後に/statuslineでステータスラインを設定し、git情報＋コンテキスト使用率＋レート制限リセット時間の2行表示で締めくくる

1
基本コマンドとPlanモードの使い方
構成
ゴール：mkdir/cd, /init, /clear, /compact, --resume, Planモードの使い分けができるようになる
重要トピック：mkdir/cdで作業ディレクトリを切る／/initでCLAUDE.md自動生成／この講座では/clearを強く推奨（作業が終わったらすぐ/clear、auto-compact入る前に自分でやる）／なぜ/clearが効くかの仕組み＝毎ターン会話全文を再送信しているので履歴が伸びるほど重くなる／コンテキストウィンドウは上限固定、/clearは「広げる」ではなく「空きを取り戻す」／副次効果で精度（context rot回避）と速度も戻る／/compactは要約を残したいときだけ／--resumeで前回セッション復旧／Planモード（Shift+Tab×2）＝計画→承認→実装／Boris氏（Claude Code開発者）も「まずPlanしてから実装」と推奨／のちのスペック駆動開発への布石
受講者のつまずき：/clearのタイミングがわからない（→作業が終わったらすぐ/clear）／/compactと/clearの使い分けで迷う（→基本/clear、残したい時だけ/compact）
画面操作方針：スライドは表紙のみ。画面収録で実演。各コマンドの実行と効果を見せる
位置付け方針：実践（基本操作の定着）
参照一覧：Boris氏（Claude Code開発者）の「まずPlanしてから実装」発言
シーンの流れ：まずmkdir/cdで作業ディレクトリを作成 → 次に/initでプロジェクトを初期化（CLAUDE.md生成）→ /clearを強く推奨する理由を解説（auto-compact前に自分でやる、毎ターン会話全文を再送信する仕組み、コンテキストの空きを取り戻す発想）→ /compactの使い所を/clearとの違いとして明示（残したい要約があるときだけ）→ --resumeで前回セッションを復旧 → 最後にPlanモード（Shift+Tab×2）で「考えてから動く」発想を実演し、Boris氏の「まずPlanしてから実装」を紹介、スペック駆動への布石で締めくくる

Claude Codeでリサーチ分析　回転寿司3社のIRを比較しよう
構成
ゴール：マーケター・経営企画・営業担当者が、回転寿司大手3社（スシロー・くら寿司・はま寿司）の公開IR資料をClaude CodeのWebFetch機能で取得し、売上高・営業利益率・既存店売上前年比の3指標を一覧化した比較マトリクスHTMLを1回の依頼で生成できるようになる。題材は誰でも知っている回転寿司業界、生成物は競合分析の実務テンプレートとして社内に持ち込める形に仕上げる
重要トピック：競合分析リサーチは複数社のIR資料をPDFで読み込んで比較表にまとめる時間消費業務（半日仕事）／Claude CodeのWebFetch機能で公開IR資料を直接取得（スシロー・くら寿司・はま寿司の決算短信URL指定）／1回のプロンプトで比較マトリクスHTML生成（売上高・営業利益率・既存店売上前年比の3指標）／これは典型的なVibe Coding（ざっくり依頼で素早く形にする、コンテキスト量が少ない、入門として最適）／公開IR資料はそもそも誰でもアクセス可能（社外秘情報とは別物、AIに読ませる心理的抵抗を解消）／マーケター・経営企画・営業の競合分析に題材を入れ替えれば応用可能／次回（次の動画）への布石（同じ題材を次回はスペック駆動で投資家向けに作り直す、コンテキスト量の差で成果物がどう変わるかの対比構造）
受講者のつまずき：「IR分析は証券アナリスト等の専門スキルが必要」という思い込み（→Claude CodeにURLと依頼文を渡せば3社比較が一発で形になる体験で解消）／「公開情報でもAIに読ませて大丈夫か」の心理的抵抗（→公開IR資料は投資家向けに公開されている情報、社外秘・個人情報とは性質が別と整理）／「3社の3指標で本当に分析になるのか」の浅さ不安（→入門として3社・3指標、実務では銘柄数や指標を増やせば段階的に深化できると位置付け）

シーンの流れ：まず冒頭で「前のセクションで3つのアプリを作りました。見た目は本物のSaaSでも、データは自分のブラウザの中だけ。ブラウザを閉じるとスマホでは何も見えないし、同僚とも共有できません。今日はその最後の壁を取り払います。名刺管理アプリにクラウドのデータベースを繋いで、『ブラウザを閉じても、別の端末でも、同じデータが見える』状態にします」と宣言（50〜60秒）→ 次にローカル保存とクラウドDBの対比を1枚のスライドで整理（ローカル＝冷蔵庫の付箋、クラウドDB＝自宅のファイルをクラウドに上げる感覚、優劣ではなく用途の使い分け）→ テーブル設計はExcelの列を決める感覚、と日常語で橋渡ししてcontactsテーブル1枚の設計を提示（id／name／company／email／phone／created_atの6列）→ Supabase管理画面のTable Editorでcontactsテーブルを作成（実演）→ 続いてプロジェクトのSettings→APIからURLとANON KEYをコピーし、ローカルに.envを新規作成して2行書く（「この1ファイルがすべての鍵、だから.gitignoreに入っていることを必ず確認」と強調）→ npm install @supabase/supabase-jsでSDKを追加、Claude Codeに「ローカルストレージの読み書きをSupabaseに置き換えて」と依頼（CRUDの4操作が一度に置き換わる様子を見せる）→ ローカルで名刺を1件登録し、Supabase管理画面のTable Editorで同じ行が増えたことを確認（「ブラウザの外側にデータが出た」瞬間）→ GitHubにpushし、Vercelの環境変数画面にURLとキーを同じように登録して本番デプロイ（.envはGitに上がらない、環境変数は本番環境にも別経路で渡す、という構造を口頭で整理）→ 本番URLをスマホで開き、さっきPCで登録した名刺がそのまま見えることを確認（「ブラウザを閉じても消えない」体験の達成点）→ 最後に「これで名刺アプリはローカル試作からクラウド本物に昇格しました。同じ流れを経費アプリ・CRMでも反復します。認証や権限管理はこの3本では扱わず、BACKSTAGE社の回でまとめて整理します」と次回予告で閉じる

経費アプリにDB接続して永続化しよう
構成
ゴール：フロント完結の経費アプリにSupabaseを接続し、経費データを端末を超えて残せるようにする。2回目の反復で「APIキー取得 → テーブル作成 → CRUD実装」という永続化の型を体得し、次に別ドメインのアプリでも同じ手順で接続できる自信を持ち帰る。
重要トピック：2回目だからリズムを上げる（前の動画で踏んだ3ステップを経費アプリで同じ順に辿り直す、再現性の体感）／別ドメインで同じ型が通用する実感（家計簿でも住所録でも同じExcelに書式だけ変えて記録する感覚）／テーブル設計の小さな拡張（expensesの単一テーブル、金額は数値型・日付は日付型・カテゴリは文字列型と列ごとに型を選ぶ発想）／ローカルストレージ → Supabaseの置き換え（既存のローカル保存ロジックを差し替えるだけでコードの骨格は変わらない）／本講座では永続化の型に集中する判断（集計クエリは組み合わせ実装可能だが、本講座ではシンプルな永続化の型を揺るぎないものにする方を優先）／レシート画像はブラウザ表示のみに留める（クラウドストレージに預ける選択肢もあるが今回は永続化の骨格に絞る）／認証・権限管理は後半に集約
受講者のつまずき：「同じ手順でもう一度やる意味が見えない」（冒頭で「2回目は手が覚えに行くフェーズ」と宣言し、速度を上げることで学習の進捗感を担保）／「金額や日付の列の型って何を選ぶの？」（金額は計算したくなるから数値型、日付は並び替えたくなるから日付型、という基準を1行で示す）／「集計グラフを外して見栄えが寂しくなるのでは」（フロント側のグラフ描画はそのまま残し、データ取得先だけDBに差し替える構成）
画面操作方針：スライド比率は低め（表紙＋完成画面キャプチャのみ、尺の1割以下）、残り9割は画面収録の一気通貫。実演順：Supabase管理画面を開く→expensesテーブル作成（列：id / date / amount / category / memo / created_at、型はそれぞれ選ぶ）→APIキーを.envに貼る（前の動画と同じ流儀で手早く）→Claude Codeに「ローカルストレージ保存をSupabaseのexpensesテーブルに置き換えて」と依頼→生成差分を確認→ブラウザで経費を登録→Supabase管理画面で行が増えたことを確認→ブラウザを閉じて開き直しても経費が残っていることを確認→別タブで同じURLを開いても同じデータが見えることで永続化を体感。個人情報配慮：経費データは架空（会議費／交通費／消耗品費など）、実在の店舗名・取引先名は使わない
位置付け方針：S6の反復定着回。前の動画で永続化の型を初めて通した受講者に、同じ型が別ドメインでも通用することを体感してもらう位置。ここで型が定着すれば、次の動画でCRMを永続化するときには手が先に動く状態に近づく。S6全体の学習曲線としては「1回目で型を学ぶ → 2回目で速度を上げて定着 → 3回目で手が覚える」の中央に置かれる回であり、受講者の自信がもっとも伸びる瞬間をここに設計する。BACKSTAGE社の回で認証・権限管理に踏み込む前に、シンプルな永続化の骨格を3つのアプリで揃えきる流れの中継点
参照一覧：前のセクションで作ったfreee風経費アプリ（ローカルストレージ版、置き換え元のコード）／Supabase公式ドキュメント https://supabase.com/docs／Supabase JavaScriptクライアント https://supabase.com/docs/reference/javascript/introduction／前のセクションのSupabaseセットアップ回（APIキーの場所と管理画面の位置が既出）／前の動画で整備した.envとSupabaseクライアント初期化パターン
シーンの流れ：まず冒頭で「前の動画で名刺管理アプリを永続化しました。今日は同じ手順を経費アプリで辿り直します。2回目なので手が覚えにいくフェーズです。完成すると、経費データが端末を超えて残るようになります」と宣言 → 次に今日の完成形を30秒プレビュー（経費を登録 → ブラウザを閉じる → 開き直しても残っている）→ Supabase管理画面でexpensesテーブルを作成（列と型を1列ずつ選ぶ、金額は数値型・日付は日付型という選び方を短く添える）→ APIキーを.envに貼る（ここは前の動画と同じ流儀で手早く通す）→ Claude Codeに「ローカルストレージ保存をSupabaseのexpensesテーブルに置き換えて」と依頼し、生成された差分を確認 → ブラウザで架空の経費を登録し、Supabase管理画面で行が増えたことを確認 → ブラウザを閉じて開き直し、経費が残っていることを体感（ここで「型が定着した」ことを言語化）→ 集計クエリにも踏み込める選択肢があるが本講座では永続化の骨格に集中すると整理し、レシート画像のクラウド保存も同じ理由で次の機会に回すと伝える → 最後に「次の動画では同じ型をCRMで3回目に辿ります。そこまで来れば永続化の型は自分の型になっています」と予告してクロージング

CRMアプリにDB接続して永続化しよう
構成
ゴール：前のセクションで作ったSalesforce風CRMにSupabaseを接続し、顧客データと商談ステータスをクラウドで永続化できるようになる。DB接続3回目の反復を通じて「もう自分で好きなSaaSのDB接続ができる」という独り立ち感覚を確立する。
重要トピック：3回目の反復で独り立ち（自転車の乗り始めは補助輪付き、2回目で補助輪を外し、3回目でスイスイ走れるようになる感覚）／customers単一テーブルで顧客×商談×パイプラインをまとめて扱う設計（id / name / company / status / last_contact / memo / created_atの7カラム、複数テーブル・外部キーを使う設計は実務では登場するが今回は扱わない）／パイプラインのステータス遷移をstatusカラム1つで実現（見込み・提案・成約の状態を単一カラムのUPDATEで切り替える、Excelの「ステータス」列書き換えと同じ）／CRUDの4動作をまとめて復習（依頼文パターンが3回目でテンプレート化）／S6前半3本の総まとめ（名刺・経費・CRMの3アプリが「ブラウザを閉じても消えない」状態に到達）／独り立ち後の景色（自分の普段使いのSaaSに置き換えて「あれも同じ手順で作れる」と言語化）
受講者のつまずき：「3回目だから手順を飛ばしても大丈夫か不安」（前の2本で身体が覚えている範囲は省略して構わない、Claude Codeへの依頼文も要点だけで通じる）／「顧客と商談を分けなくて本当に大丈夫か」（今回はシンプルにデータが残ることが目的、単一テーブルで十分機能する、必要になった段階で分ければいい）／「パイプラインのドラッグ操作がDB反映されるか心配」（ステータスが変わるたびにstatusカラムをUPDATEするだけ、画面の見た目とDBの中身は同じことの2つの表現）
画面操作方針：スライドは表紙＋S6前半3本の3アプリ完成後の全体像図のみ、画面収録で実演。Supabase側でcustomersテーブル作成→前のセクションのCRMコードを開く→ローカルストレージ保存箇所をSupabaseクライアントに置き換えるようClaude Codeに依頼→CRUD動作確認（顧客追加／一覧表示／ステータス変更／削除）→ブラウザを閉じて再度開きデータが残っていることを確認→GitHub push→Vercel自動デプロイ。最後に名刺・経費・CRMの3アプリをブラウザタブで並べて、それぞれブラウザを閉じても消えない状態を見せるツアーで締める。スライドと画面収録の比率は1:9。顧客データは架空（「株式会社サンプル商事」「見込み客A」程度）、実在企業・個人名は使わない
位置付け方針：S6の前半3本（DB接続編）の総まとめ回。前の2本で身につけた「テーブル設計 → Supabase接続 → CRUD置き換え」のフローを3回目で完全に定着させ、受講者が自分一人で好きなSaaSにDB接続できる独り立ち感覚を持ち帰る回。前のセクションで作ってきた3つのフロントアプリが全て永続化され、ローカルストレージ時代の「ブラウザを閉じたら消える」制約を卒業するマイルストーンになる。この総まとめを踏まえて、次の2本でプロの開発組織のリアルなClaude Code活用フローを見て、今の自分の到達点がプロの現場とどう地続きになっているかを比較する展望に繋ぐ
参照一覧：Supabase公式 https://supabase.com/／Supabase JavaScript Client https://supabase.com/docs/reference/javascript/introduction／前のセクションで作ったSalesforce風CRMのコード（ローカルストレージ版、ベースとなるアプリ）／前2本のDB接続回（名刺・経費）で整備した.envとSupabaseクライアントの初期化パターン／前のセクションで扱った「秘密情報の扱い」回（APIキーの環境変数管理の復習）
シーンの流れ：まず冒頭で「前の2本で名刺と経費をクラウドに保存できるようにしました、残るは前のセクションの最後に作ったSalesforce風CRMです、3回目の今回はサクッと進めて、見終わる頃には自分の普段使いのSaaSにもDB接続できる手応えを持ち帰ってもらいます」と宣言し、3回目の反復で独り立ち感覚を確立するのが今日の目的だと腰を据えて語る → 次にSupabase側でcustomersテーブルを作る、カラムはid / name / company / status / last_contact / memo / created_atの7つ、「顧客と商談を分けない、パイプラインのステータスもstatusカラム1つで表現する、シンプルにデータが残ることを優先する」という設計判断を1文で言い切る → 前のセクションで作ったCRMのコードを開き、ローカルストレージ保存の箇所をSupabaseクライアントへのCRUD呼び出しに置き換えるようClaude Codeに依頼、3回目なので依頼文が要点だけで通じることを体感する → ブラウザで顧客を1件追加し、SupabaseのTable Editorにレコードが届いていることを確認 → パイプラインのカードをドラッグして「見込み」から「提案」へ動かし、statusカラムがUPDATEされたことをTable Editorで確認、画面の見た目とDBの中身が同じことの2つの表現であることを腹落ちさせる → 顧客を1件削除し、画面とDBの両方から消えることを確認、ブラウザを閉じて再度開いても他のデータは残り続ける様子を見せる → GitHubにpushしてVercelが自動デプロイ、発行URLにスマホからアクセスして、端末を超えて同じデータが見える状態を確認 → 最後に名刺・経費・CRMの3つをブラウザタブで並べて、それぞれブラウザを閉じても消えない状態を順にツアーし、「前の2本と今回で、前のセクションで作った3つのSaaS風アプリが全て永続化できました、ここまで来たら自分の普段使いのSaaSにも同じ手順でDB接続できます、次の2本ではプロの開発組織のリアルなClaude Code活用フローを見て、今日たどり着いたこの到達点がプロの現場とどう地続きになっているかを見ていきます」と展望を語って締めくくる
