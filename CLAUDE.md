# Claude Code実践講座 プロジェクトルール

## 作業モード判断（reasoning effortの切り替え提案）
- 作業の性質が変わるタイミングで、effort（high / max）の切り替えを土屋さんに**こちらから提案する**
- **highで十分**：お手本パターン応用・構造化テンプレート埋め・バッチ進行（例：scripts-data.jsへの台本書き起こし、構成7項目の機械的反映、検証＆コミット）
- **max推奨**：ゼロベースの構成検討・アーキテクチャ判断・ルール設計・複雑なデバッグ・初見ドメインの判断（例：S6-V4/V5構成検討、S5アプリ架構、CLAUDE.md／メモリのルール改定、外部API設計レビュー）
- 提案フォーマット：「次の作業は◯◯（性質：◯◯）なので、effort △△ に切り替えるのを推奨します」と1〜2文で

## WBS・ガントチャート管理
- WBS・ガントチャートの唯一の情報源は `docs/curriculum-review.html`（HTMLレビューサイト）
- タスクの追加・削除・変更はすべてこのファイルに対して行う
- `docs/gantt-chart.md` は使わない

## 収録環境（Claude Code Desktop版）

### 前提
- 教材全体でClaude Code Desktop版を使う（カーソル/IDE統合は使わない）
- Desktop版は1ウィンドウで完結する2ペイン構成
  - **中央：チャットUI**（対話・コード生成・ファイル編集）
  - **右：内蔵ターミナル**（`claude`コマンドでCLI版を起動するとフル機能）

### 管理系コマンドの扱い
- Desktop版チャットUIでは使えないコマンドがある：`/plugin` `/mcp` `/reload-plugins` `/skill` `/hooks` 等の**管理系**
- これらは**右の内蔵ターミナルで `claude` を起動**してから実行する
- MCPプラグイン（Notion/Figma/Slack/claude-mem等）の**利用自体**はDesktop版チャットUIからも可能（インストール済みなら自動接続）

### 教材での受講者学習動線
- **S1冒頭**で「2ペイン構成」を紹介（カーソル/IDEではなくDesktop版を使う理由含む）
- **S1〜S2（入門）**：中央チャットUIだけで完結
- **S3以降**：必要に応じて右ターミナルに `claude` を起動して管理系コマンドを扱う

### HTMLスライドの仕様
- **1動画＝1HTMLファイル**（縦スクロール形式）
- 既存のPoCデモ（`demos/s5-poc/` 屋上防水SEO等）のHTMLレポート構造を流用
- スライドごとにClaude Code Desktop操作が必要な箇所は、画面収録に切り替える

## 制作パイプライン全体像

### フロー
レビューサイト（構成＋台本）→ スプレッドシートに転記 → カリキュラム進捗管理シート → 収録・編集 → LMS公開

### 1. レビューサイト（`docs/curriculum-review.html`）— Single Source of Truth
- 役割：各レクチャーの**構成と台本**を一元管理する唯一の情報源
- 構成に必要な7項目（これが揃えば台本が書ける）：
  1. **ゴール** — 見終わった受講者が何ができるようになるか
  2. **重要トピック** — カバーする内容
  3. **受講者のつまずき** — 予想される詰まりポイント
  4. **画面操作方針** — デモあり/なし、何を見せるか
  5. **位置付け方針** — 導入/実践/まとめ等
  6. **参照一覧** — 使う資料・CSV・URL・デモファイル等
  7. **シーンの流れ** — 話題の順序（「まず〜／次に〜／最後に〜」の話し言葉で繋ぐ。Step番号は使わない）
- 台本は `docs/scripts-data.js` に記述し、`curriculum-review.html` から動的レンダリング（関心の分離）
  - 構成7項目はHTML側（`curriculum-review.html`）に直書き
  - 台本本体は `scripts-data.js` の `SCRIPTS` オブジェクトに追加
  - 1エントリの構造：
    - `meta`：duration / mode / goal
    - `materials[]`：type / name / purpose / timing
    - `script[]`：start / end / topic / direction / content / reference

#### scripts-data.js エントリの作法
- **duration 表記**：「約◯分◯秒」形式で統一する（「約6:30」のコロン区切りは不可）。きっかりn分の場合は「約◯分」も可
- **materials[].timing**：内容ベース表現を使う（「冒頭・クロージング」「序盤」「中盤」「中盤〜後半」「後半の動作確認」「クロージング直前」等）。「Step X」「シーン◯」の番号表記は禁止
- **対象者宣言**：オープニング0:00-0:20以内に「この動画では〜の方に向けて」と受講者像を明示する（S1-V4が基準）
- **冒頭の「なぜ扱うか」**：0:20-1:00の40〜60秒で1シーンにまとめる。2シーンに分散しない。**「作業を削減する」だけで終わらせず、「その結果として何が良くなるか（本質的なビジネスメリット：現状把握・意思決定の質・問題発見スピードなど）」まで書き切る**
- **cps（文字/秒）の目安**：各シーン **3.0〜4.5 cps** が落ち着いたナレーションの帯。**5.0 超は詰め込みすぎのサイン**。台本完成後に Python で計測（HTMLタグを除いた plain text 文字数 ÷ 秒）してペースを均す。オープニングなど短尺シーンは帯を外れやすいので、該当シーンを延ばす or 内容を削る
- **自セクション参照**：「前回の動画で扱いました」「次の動画で扱います」は可。セクション大枠（「S5でアプリ開発を扱います」）も可。動画番号（S2-V2／S1-V4 等）の固定指定は避ける（構成変更で陳腐化するため）
- **カジュアル表現の置換**：「〜ですよね」（押し付け）→「〜と思います」／「でも」→「一方で」／「〜じゃなく」→「〜ではなく」／「〜って言った」→「〜とお伝えした」／「〜みたいな」→「〜のような」／「ちゃんと」→「しっかり」／「食う」（トークン消費）→「消費する」
- **お手本**：S1-V4（`営業マン向け自動見積り作成ツール`）の scripts-data.js 該当エントリを、新規台本執筆前に参照する

#### HTML／scripts-data.js 書き換えの作法
- `curriculum-review.html` と `scripts-data.js` は**どちらも全角括弧「（）」を多用**している（日本語コンテンツのため、topic名・content本文に混在）
- `Edit` ツールは文字列の完全一致で動くので、半角「()」と取り違えると失敗する
- **1セクション分以上（video要素2つ以上 or scripts-data.js の1動画まるごと）の書き換えは Python スクリプトで実施**：
  1. `/tmp/rewrite_xxx.py` に置換スクリプトを書く（triple-quoted文字列で括弧を素直に扱える）
  2. Bashで実行
  3. `git diff` で確認
  4. スクリプト削除
- 小さな1〜2箇所の修正は `Edit` でOK。ただし**old_stringは必ずReadで取得したテキストをそのままコピペ**（手入力すると括弧の取り違え起きる）

#### 台本修正時の構成7項目 同期ルール
- 台本（`scripts-data.js`）を修正した後、構成7項目（`curriculum-review.html`）も**台本に合わせて同期する**
- 同期した結果、**他動画と構成粒度・項目の厚み・シーン数が揃わなくなっても問題ない**（各動画の実態を優先した結果）
- 他動画と比較するのではなく、**台本との一致**を優先する
- 同期したことを示すため、該当 video要素の `.video-structure` 直下に `<div class="sync-note">※台本（scripts-data.js）に合わせてYYYY-MM-DDに構成7項目を同期。他動画と構成粒度や項目の厚みが異なる場合があります</div>` を入れる

### 動画タイトル（共通ルール）
- **35文字以内**（LMS表示制約、超えると末尾が切れる）
- 形式：**「【職種向け／業務種別向け】本題」**（副題「— ◯◯」を入れる場合も全体で35字以内に収める）
- 語尾：**「〜しよう」を推奨**（「〜を自作しよう」「〜を自動化しよう」「〜を生成しよう」など受講者への呼びかけ形で統一）
- 数え方：全角・半角・記号・スペース含め1文字ずつカウント
- 例：「【人事向け】社内eラーニングシステムを自作しよう」（24字）

### 構成レビューチェックリスト（完成度を高める評価軸）

構成7項目を「埋める」だけでなく、各項目の質を以下の観点で点検する。

#### 必須評価軸（どの動画にも適用）
- **ゴール**：動詞＋目的語＋アウトプット形が具体／学習目的が1文で明示されている
- **重要トピック**：各トピックに「日常との橋渡し」「中立化（断罪表現なし）」「具体例」が組み込まれている
- **受講者のつまずき**：体験ベース（推測でない）／2〜3個に絞る／ゴールと直結
- **画面操作方針**：スライドと画面収録の比率が決まっている／実演手順がコマンド連鎖で具体的／個人情報（マイナンバー等）の扱いが明記されている
- **位置付け方針**：単なるラベル（実践／導入）でなく**戦略文**になっている（章全体での位置・次動画への接続が書かれている）
- **参照一覧**：必要素材（過去動画／引用元／既存ファイル）が具体パス・URLで揃っている／**技術系の動画は使うパッケージ・API が deprecated/非推奨になっていないか直近1年を WebSearch で確認**（MCP / npm / SDK / プラグイン系は変動が激しく、古い手順を教えてしまうリスクが高い）
- **シーンの流れ**：冒頭50〜60秒で「なぜ重要か」を宣言／同じメッセージを繰り返さない／矢印繋ぎの散文で記載（「まず」「次に」は各1回、以降は接続詞なし）／9〜12シーン

#### 動画間の接続言及ルール
- **他動画への具体的な番号指定（例：「S5-V6で扱う」「S3-V2と連動」）は避ける**
- 番号は構成変更で変動する／本質でない接続情報は受講者の混乱の元
- 「S5でアプリ開発を扱います」「次セクションでさらに深掘りします」のような**大枠言及にとどめる**
- トピック自体への言及（「freee風アプリは別の回で扱います」）はOK、動画番号の固定指定がNG

#### 設計の引き出し（使える場面で使う、必須ではない）
- **既知題材の再利用**（S2-V5型）：過去動画のアウトプットを別観点で作り直す
- **対比構造**：A vs B を並べて学習効果を倍化（例：Vibe版 vs Spec版、Skillなし vs Skillあり）
- **次セクションへの布石**：連続性を作る（例：「これは次セクションでさらに深掘りします」※動画番号の固定指定はしない）

#### お手本
- S2-V5（`スペック駆動開発を実践しよう`）／S3-V1（`確定申告Skillで学ぶSkillsの極意`）の `docs/curriculum-review.html` 該当箇所を、新規構成作成前に参照する

### 台本作成ワークフロー（実演デモ講座向け）
- 土屋が Screen Studio で画面収録＋ラフ解説
- Screen Studio のトランスクリプト機能で文字起こし
- Claude Code（/daihon Skill）で川端さん用台本に整形
- 川端さんがナレーション再収録
- 編集チームが映像＋ナレーションを合成

### 台本執筆スタイル（川端さんナレーション用）

#### トーン：中級者向けの落ち着いた口語体
- カジュアル過ぎず、硬すぎない。中級者が一度で腹落ちするペース
- 川端さん口調「〜ですね」「〜しましょう」「やってみます」を基調
- 押し付け表現（「〜ですよね」）は避け、「〜と思います」など柔らかく

#### 推奨語と避ける語
- 「やります」→「扱います／進めていきます」
- 「投げる」→「依頼する」
- 「なるんです」→「になります／ようになります」
- 「でも」→「一方で」
- 「そんなの無理」→「現実には不可能」
- 「突っ込む」→「着手する」
- 「丈夫さ」→「堅牢さ」

#### 見出しの作法
- 「Step 1：」「Step 2：」の**見出し番号は使わない**
- 「まず〜」「次に〜」「最後に〜」で話し言葉として繋ぐ
- 構成7項目の「シーンの流れ」も同じ作法で書く

#### 中立化（断罪表現を避ける）
- 手法比較（例：Vibe Coding vs Spec-Driving）は**優劣ではなく「コンテキスト量の差」「用途の使い分け」**で整理
- 「落とし穴」→「限界」
- 「破綻します」→「狙い通りが出にくくなります」
- 「ダメ」「悪い」は避け、特性で説明

#### 専門用語と日常語の橋渡し
- カタカナ専門用語（例：「スペック」）は**日常での使われ方（例：「パソコンのスペック」）との接続**を1文で入れる
- 英語略称は**英語フルスペル＋日本語訳**を添える（例：spec = specification = 仕様書）

#### 論理構造
- 冒頭シーンで**「なぜ扱うのか／メリットは何か」**を腰を据えて語る（目安50〜60秒）
- 同じメッセージを複数シーンで繰り返さない（冒頭で宣言→後続シーンは定義・実演に絞る）

#### 題材スコープの階層
- **S1・S2（入門）**：HTMLレポート単品レベル。レビュー・テスト・動作検証は扱わない
- **S5・S6（アプリ開発）**：DB・CI/CD・デプロイ・テスト戦略まで扱う

### 2. 台本スプレッドシート（Googleスプレッドシート）
- 役割：レビューサイトからの**転記先**。編集チーム・音声担当への配布用
- レビューサイトの台本と内容は同一。スプレッドシート側で独自に書き換えない
- 実例：M5セクション1動画1

### 3. カリキュラム進捗管理シート（Googleスプレッドシート）
- 役割：動画編集チームとの**制作進捗管理**
- カリキュラム進捗管理シートの列構造（カテゴリ別）：
  - **動画メタ情報**: コース名 / 時間（分） / 画面収録（分） / 動画タイトル / 動画の説明文 / 需要度 / 重要度 / 難易度 / セキュリティ / 職種メインカテゴリ / クイズ内容 / 講師担当
  - **担当者**: ヒルウラ担当 / 外部編集担当 / 音声担当 / FB優先順位 / プロマネ
  - **ステータス管理**: ステータス / FB / ステータス更新日時 / Youtubeリンク
  - **素材データ**: 修正用データ / スライド / アバター映像 / 画面収録 / ヒルウラ編集後データ（mp3）
  - **編集パイプライン**: 編集着手 → 初稿完了予定 → 黒河FB → スライドデータ準備 → 音声+アバター準備 → 画面収録データ準備 → BSデータ準備 → 編集着手 → 納品 → 黒河FB → 川端FB → 最終版提出 → 川端最終チェック → LMS公開予定日
  - **台本・資料**: 担当 / A-E列完了 / 台本完了 / スライド数
  - **撮影**: 担当 / ステータス / 成果物
  - **最終チェック**: 担当 / 動画時間 / 動画本数 / 成果物
  - **進捗フラグ**: 目次 / 資料作成 / 台本 / 撮影 / 編集 / チェック / 納品
  - **備考**: 音声データ不備 / 川端FB反映有無
